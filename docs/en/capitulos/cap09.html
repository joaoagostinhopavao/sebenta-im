9  FHIR: Inter-Hospital Interoperability

Introduction

The previous chapter built the FHIR vocabulary: what a Resource is, how identifiers work, what primitive and composite types are, what a CodeableConcept is, and how a Bundle acts as an envelope. That vocabulary is necessary — but not sufficient. Knowing what a Patient or an Observation is does not explain how those resources connect to model a real clinical workflow, nor how that connection supports the exchange of information between different institutions.

This chapter uses the vocabulary to build workflows. The focus shifts from the individual structure of each resource to the network of relationships that links them: how an Encounter articulates Patient, Practitioner and Organization; how a DiagnosticReport aggregates Observation; how an ImagingStudy references DICOM images without duplicating them. And, finally, how that graph of resources can be transported between different servers — with its guarantees and its real limitations.

9.1 From Structure to Flow

The starting point is a transition table: what was covered in the previous chapter and what this chapter adds to each concept.

The central idea of this chapter can be summarised in one sentence: interoperability is not merely message exchange — it is a graph of identified, referenced and standardised clinical entities. The same patient exists in multiple systems; FHIR does not duplicate the data, it links them.

9.2 Reference() — How Resources Connect

9.2.1 Anatomy of a Reference()

A Reference() is a typed pointer to another resource — local, on another server, or embedded within the resource itself. It does not copy the data; it points to it. The client that needs the data resolves the reference with a GET to the target URL.

A Reference() has three fields (HL7 International, 2019):

The display field is for presentation only — if the system updates the patient’s name in the Patient resource, the display field in any Reference() pointing to that patient is not updated automatically. The canonical data always resides in the referenced resource, not in the reference.

{
  "resourceType": "DiagnosticReport",
  "id":           "report-987",
  "status":       "final",
  "code":         { "text": "Full blood count" },
  "subject": {
    "reference": "Patient/123",
    "type":      "Patient",
    "display":   "Ana Ferreira"
  },
  "performer": [{
    "reference": "Practitioner/dr-costa",
    "type":      "Practitioner",
    "display":   "Dr. Luís Costa"
  }],
  "result": [
    { "reference": "Observation/obs-1" },
    { "reference": "Observation/obs-2" }
  ]
}

In this DiagnosticReport, the subject field points to Patient/123 — the Patient is not copied, it is referenced. If a client needs the complete patient data, it issues GET /Patient/123.

9.2.2 Reference Types and Resolution

FHIR defines four ways to reference a resource, with distinct implications for portability and network dependency (HL7 International, 2019):

Absolute reference — includes the full server URL. Portable across servers, but requires the target server to be accessible at resolution time:

"subject": { "reference": "https://fhir.ulstmad.pt/Patient/123" }

Relative reference — the most common. The URL is relative to the base server of the transaction. Simple and efficient, but only works within the context of the same FHIR server:

"subject": { "reference": "Patient/123" }

Contained resource — the referenced resource is embedded within the parent resource, with no URL of its own. The # prefix identifies the reference as internal to the current resource:

"subject": { "reference": "#patient-inline" }

Logical reference — carries only the clinical identifier (business identifier), without pointing to any server. Resolution is left to an external system (e.g. a Master Patient Index):

"subject": {
  "identifier": {
    "system": "http://rnsnp.min-saude.pt",
    "value":  "123456789"
  }
}
Note

Note — Relative references are the default choice in intra-hospital environments where all resources reside on the same server. Absolute references are necessary in inter-hospital scenarios. Logical references are the solution when there is no guarantee that the target server is accessible — the clinical identifier travels; resolution comes later.

9.3 Bundle — Types and Uses

The Bundle is the FHIR standardised envelope. It allows a set of resources to be transported as a coherent unit. The five Bundle types have distinct semantics:

The most important distinction for interoperability scenarios is between transaction and batch. In a Bundle transaction, either all operations succeed or none is persisted — an atomic guarantee essential when data consistency cannot be partial. In a Bundle batch, each request is processed independently: one may fail without affecting the rest.

The document type is structurally different from the others: the first entry element must be a Composition (the document header resource, with date, author and sections), and the resulting Bundle is considered immutable by convention — it represents a deliberate, signed snapshot of the clinical state at a given moment.

9.4 Clinical Workflow Resources

The resources covered in this chapter form the backbone of any clinical episode: Patient (who is treated), Practitioner/PractitionerRole (who treats and in what role), Organization (where), Encounter (the episode that articulates them all), DiagnosticReport and Observation (the results), and ImagingStudy (the bridge to the PACS).

9.4.1 Patient — the Subject of All Care

Patient is the most referenced FHIR resource. All other clinical resources — Encounter, Observation, DiagnosticReport, MedicationRequest — point to it. The fields most relevant to interoperability:

The identifier field is the key to interoperability: the same patient exists in multiple systems with different system ids, but the national health number or the NIF is the same in all. Interoperability begins by correctly identifying the patient in each system.

{
  "resourceType": "Patient",
  "id":           "p-123",
  "identifier": [
    { "system": "http://rnsnp.min-saude.pt",    "value": "999000111"       },
    { "system": "http://ulstmad.pt/processos", "value": "PROC-2024-04512" }
  ],
  "name":        [{ "family": "Ferreira", "given": ["Ana"] }],
  "birthDate":   "1978-04-12",
  "gender":      "female",
  "managingOrganization": { "reference": "Organization/org-ulstmad" }
}

9.4.2 Practitioner and PractitionerRole — Identity and Role

FHIR deliberately separates the identity of a healthcare professional from the role they perform in a given institutional context. This separation is necessary because the same professional may hold different roles in different organisations: a cardiologist at the Unidade Local de Saúde de Trás-os-Montes e Alto Douro (ULSTMAD) and a consultant at a private hospital, for example.

Practitioner — who they are:

Field Content
identifier Professional registration number, NIF
name Full structured name
telecom Professional contact
qualification Specialty and certifications

PractitionerRole — where and in what role:

Field Content
practitioner Reference(Practitioner) — who they are
organization Reference(Organization) — where they work
code Role: physician, nurse, pharmacist…
specialty Specialty: cardiology, radiology… 
/* Practitioner — the permanent identity */
{
  "resourceType": "Practitioner",
  "id":           "pr-luiscosta",
  "identifier": [{ "system": "http://ordemmedicos.pt", "value": "C-45231" }],
  "name":        [{ "family": "Costa", "given": ["Luís"] }]
}
/* PractitionerRole — the role at ULSTMAD */
{
  "resourceType":  "PractitionerRole",
  "id":            "role-luiscosta-ulstmad",
  "practitioner":  { "reference": "Practitioner/pr-luiscosta" },
  "organization":  { "reference": "Organization/org-ulstmad" },
  "code":      [{ "coding": [{ "code": "doctor" }] }],
  "specialty": [{
    "coding": [{
      "system":  "http://snomed.info/sct",
      "code":    "394579002",
      "display": "Cardiology"
    }]
  }]
}

Separating identity from role allows the same professional to appear with different roles in different contexts without duplicating their personal data and credentials. Dr. Luís Costa is a Cardiologist (specialty) at ULSTMAD (organization) — PractitionerRole links all three.

9.4.3 Organization — the Institution as a Resource

Organization represents any formal healthcare entity: hospital, department, pharmacy, insurer. The partOf field allows hierarchies to be represented — a cardiology department is partOf the hospital that houses it:

Organization is referenced by Patient.managingOrganization, Encounter.serviceProvider and PractitionerRole.organization — it is the institutional anchor field of almost all other clinical resources.

9.4.4 Encounter — the Clinical Episode as a Linking Node

Encounter represents a contact between the patient and the healthcare system — a consultation, an inpatient admission, an emergency visit, a teleconsultation. It is the resource that articulates all others in a clinical episode.

Encounter is the central node of the clinical graph: it articulates who (Patient), who treats (Practitioner), where (Organization) and when (period). All clinical results of an episode — Observation, DiagnosticReport, MedicationRequest — reference the Encounter to place the information in its temporal and institutional context.

{
  "resourceType": "Encounter",
  "id":           "enc-2025-04152",
  "status":       "finished",
  "class":        { "code": "outpatient" },
  "subject":         { "reference": "Patient/p-123" },
  "participant": [{ "individual": { "reference": "Practitioner/pr-luiscosta" } }],
  "serviceProvider": { "reference": "Organization/org-ulstmad" },
  "period": {
    "start": "2025-04-15T09:00:00+01:00",
    "end":   "2025-04-15T09:45:00+01:00"
  },
  "reasonCode": [{
    "coding": [{
      "system":  "http://snomed.info/sct",
      "code":    "44054006",
      "display": "Type 2 diabetes mellitus"
    }]
  }]
}

9.4.5 DiagnosticReport and Observation — Structured Clinical Result

DiagnosticReport aggregates individual Observations. A laboratory report — a full blood count, for example — is a DiagnosticReport with a list of Observations, one per measured value.

DiagnosticReport — the report:

Field Content
code Exam type (LOINC) — e.g. "Full blood count"
status registered | partial | final | amended | entered-in-error
subject Reference(Patient)
performer Reference(Practitioner \| Organization)
result List of Reference(Observation)
conclusion Narrative conclusion text

Observation — the individual measurement:

Field Content
code What was measured (LOINC) — e.g. "Haemoglobin"
value[x] The value: valueQuantity, valueString, valueCodeableConcept
status final | preliminary | amended | cancelled
subject Reference(Patient)
encounter Reference(Encounter) — clinical context
referenceRange Reference range for interpretation

The value[x] field uses the [x] notation to indicate polymorphism: the value type may be valueQuantity (e.g. 6.4 mmol/L), valueString (e.g. a textual description), or valueCodeableConcept (e.g. "positive" coded in SNOMED). Only one type may be present in a given resource.

{
  "resourceType": "DiagnosticReport",
  "id":           "dr-fbc-2025",
  "status":       "final",
  "code": {
    "coding": [{
      "system":  "http://loinc.org",
      "code":    "58410-2",
      "display": "CBC panel - Blood by Automated count"
    }],
    "text": "Full blood count"
  },
  "subject":   { "reference": "Patient/p-123" },
  "encounter": { "reference": "Encounter/enc-2025-04152" },
  "performer": [{ "reference": "Organization/lab-ulstmad" }],
  "result": [
    { "reference": "Observation/obs-hgb-001" },
    { "reference": "Observation/obs-plt-001" }
  ],
  "conclusion": "Full blood count within normal parameters."
}
{
  "resourceType":   "Observation",
  "id":             "obs-hgb-001",
  "status":         "final",
  "code": {
    "coding": [{
      "system":  "http://loinc.org",
      "code":    "718-7",
      "display": "Hemoglobin [Mass/volume] in Blood"
    }],
    "text": "Haemoglobin"
  },
  "subject":   { "reference": "Patient/p-123" },
  "encounter": { "reference": "Encounter/enc-2025-04152" },
  "valueQuantity": {
    "value": 13.8, "unit": "g/dL",
    "system": "http://unitsofmeasure.org"
  },
  "referenceRange": [{
    "low":  { "value": 12, "unit": "g/dL" },
    "high": { "value": 16, "unit": "g/dL" }
  }]
}

The DiagnosticReport.result field is a list of references — each Observation can be accessed and interpreted individually. This granularity allows a clinical alerting system to access the haemoglobin Observation directly without downloading the complete report.

9.4.6 ImagingStudy — the DICOM↔︎FHIR Bridge

ImagingStudy represents a DICOM study in the FHIR model without duplicating the images — these remain in the PACS. The resource acts as a pointer: it contains the study metadata and a reference to the WADO-RS server where the images can be accessed.

The subject field requires particular attention: the DICOM Patient ID does not map directly to the system id of a FHIR Patient resource. The two systems use independent identification schemes; identity resolution — the process of determining which FHIR Patient corresponds to a given DICOM Patient ID — is performed using matching algorithms based on NIF, name and date of birth, or by an MPI.

Note

Note — WADO-RS (Web Access to DICOM Objects — RESTful Services) is the DICOMweb protocol that exposes DICOM studies via an HTTP API. A GET request to the WADO-RS URL of a study returns the images in DICOM or JPEG format. ImagingStudy in FHIR stores this URL in the endpoint field, allowing any FHIR application to access PACS images without needing to know the details of the legacy DICOM protocol.

9.5 Distributed Interoperability

9.5.1 Scenario 1 — Referencing Across Institutions

In this scenario, Hospital A is the Unidade Local de Saúde de Trás-os-Montes e Alto Douro (ULSTMAD) and Hospital B is the Centro Hospitalar Universitário de São João (CHUSJ).

Hospital B (CHUSJ) creates a DiagnosticReport for a patient registered at Hospital A (ULSTMAD). The Patient data is not copied — it is referenced by the absolute URL of Hospital A’s server:

{
  "resourceType": "DiagnosticReport",
  "id":           "dr-chusj-001",
  "status":       "final",
  "subject": {
    "reference": "https://fhir.ulstmad.pt/Patient/123",
    "display":   "Ana Ferreira"
  },
  "code":      { "text": "Chest CT" },
  "performer": [{ "reference": "Organization/org-chusj" }]
}

When Hospital B needs the complete patient data, it resolves the reference with a GET:

GET https://fhir.ulstmad.pt/Patient/123
→ Hospital A responds with the complete Patient resource

This architecture has a real limitation: if Hospital A’s server is not accessible at the time of the request, the absolute reference cannot be resolved. FHIR does not guarantee availability — it is the responsibility of the institutions to ensure SLA agreements and high-availability infrastructure.

9.5.2 Scenario 2 — Transfer and Continuity of Care

Hospital A will transfer a patient to Hospital B. The flow uses a Bundle of type document.

Step 1 — Hospital A creates the Bundle(document):

It includes a Composition (mandatory header — date, author, sections), the Patient, the discharge Encounter, the DiagnosticReport, and the MedicationRequest. The Bundle document is immutable by convention — it represents a deliberate, signed snapshot of the clinical state at the moment of discharge.

Step 2 — Transmission via authenticated POST:

POST https://fhir.chusj.pt/Bundle
Authorization: Bearer <token OAuth2>

The POST is authenticated via OAuth2/SMART on FHIR — required in production by the GDPR (Regulation (EU) 2016/679, Art. 32). The Bundle is stored at both institutions: one copy at Hospital A, another at Hospital B.

Step 3 — Hospital B receives and processes:

Hospital B stores the Bundle and resolves the patient identity using the shared identifier (NIF):

  • If the patient already existed locally → associates without duplication.
  • If new → creates a local Patient with the NIF as anchor, and retains the absolute reference to Hospital A as an additional identifier.

Step 4 — Continuity established:

The physician at Hospital B consults the DiagnosticReport and the MedicationRequest from Hospital A without any manual intervention. FHIR works like the web: each resource has its URL, and other systems reference it by URL. The Bundle(document) is the equivalent of saving the page to read offline — a deliberate snapshot for when the connection cannot be relied upon.

9.7 Real Limitations of FHIR Interoperability

Technical interoperability is a necessary condition — but not sufficient. The real challenges of FHIR interoperability in production are primarily semantic and organisational.

9.7.1 Patient Identity

The same patient has different IDs at each institution. Without an MPI (Master Patient Index) or a matching algorithm, cross-references cannot be safely resolved. In Portugal, the national health number (SNS) serves as the national identifier — but not all systems record it consistently. The NIF is an alternative, but raises privacy concerns.

9.7.2 Resource Versioning

A reference points to the current version of the resource — but data changes. The meta.versionId field and the _history endpoint allow access to previous versions:

GET /Patient/p-123/_history/2    ← specific version
GET /Patient/p-123/_history      ← complete history

These mechanisms require the FHIR server to implement them — and many production servers do not do so completely.

9.7.3 Terminology Conflicts

Two systems may use different terminologies for the same clinical concept. FHIR recommends — but does not mandate — the use of SNOMED CT, LOINC and ICD as standard terminologies. The ConceptMap resource allows explicit mappings to be defined between two code systems, but keeping those mappings up to date requires continuous organisational effort.

9.7.4 Authorisation and Privacy

Accessing a resource on another server requires authentication and authorisation. SMART on FHIR defines the OAuth2 flow for clinical applications — the mechanism that allows an application to request authorisation from a FHIR server to access a patient’s data, with the patient’s explicit consent. IHE ATNA (Audit Trail and Node Authentication) defines how access to clinical data is logged and audited. Without these mechanisms, sharing clinical data between institutions violates the GDPR.

1. A private laboratory receives a blood sample with the patient’s NIF but without access to the hospital’s FHIR server where the patient is registered.

  1. Which FHIR mechanism allows the laboratory to include minimal patient information in the result without needing a resolvable reference?
  2. Which reference type does the laboratory use in the subject field of the Observation?
  3. When the result reaches the hospital server, how does the hospital associate the Observation with the correct Patient?

2. Analyse the following Encounter fragment:

{
  "resourceType": "Encounter",
  "id":     "enc-emergency-001",
  "status": "in-progress",
  "class":  { "code": "emergency" },
  "subject":         { "reference": "Patient/p-789" },
  "participant": [{ "individual": { "reference": "Practitioner/pr-234" } }],
  "serviceProvider": { "reference": "Organization/org-hgv" },
  "period":          { "start": "2025-11-03T02:15:00+00:00" }
}
  1. What state is the episode in? What type of contact does it represent?
  2. The period field has no end. What does this mean clinically?
  3. What would be the URL to retrieve the Practitioner treating the patient?
  4. If a blood pressure Observation is recorded during this episode, what reference must it have in the encounter field?

3. Explain the difference between Practitioner and PractitionerRole and justify why FHIR separates the two concepts into distinct resources. Give an example with a professional working in two different contexts.

4. A Portuguese hospital prepares a Bundle of type document to send a patient’s clinical record to a private clinic.

  1. Which resource must be the first entry in the Bundle document? What is its role?
  2. The Bundle is POSTed without authentication. Which European regulation is potentially violated?
  3. After receiving the Bundle, the clinic finds that the patient already exists locally with a different id. Which Patient field should be used to confirm they are the same person?
  4. Which property of the Bundle document ensures that the received data cannot be altered retroactively?

5. Consider two systems communicating via FHIR. System A uses SNOMED CT to code diagnoses; System B uses ICD-10-CM.

  1. Which FHIR resource type allows the mapping between the two code systems to be defined?
  2. The code field of a Condition has a required binding to an ICD-10 ValueSet. System A attempts to send a diagnosis coded only in SNOMED CT. What happens during validation?
  3. Which FHIR structure would allow the diagnosis to be carried in both terminologies simultaneously?

6. A reference in an Encounter points to "https://fhir.ulstmad.pt/Patient/p-123". The ULSTMAD server is temporarily unavailable.

  1. What type of reference is this?
  2. What impact does the server’s unavailability have on the ability to process the Encounter?
  3. Which alternative reference type could make the Encounter processable even without a connection to ULSTMAD? What would its limitations be?
