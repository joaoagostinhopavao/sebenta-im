10  IHE: Integration Profiles in Healthcare

Introduction

The previous chapter demonstrated how FHIR models clinical entities and transports them between servers: Reference() to link resources, Bundle to package workflows, and SMART on FHIR to authorise access. That framework answers the question of what to transport and in what format. It does not answer the question of how to coordinate: who sends what, to whom, when, in what order, and with what conformance guarantees.

This is the function of IHE — Integrating the Healthcare Enterprise. IHE does not define new data formats: it uses existing standards (HL7, DICOM, FHIR) and specifies integration profiles — concrete scripts that determine which systems are involved, which messages they exchange, and in what sequence. An IHE profile transforms a generic standard into a testable operational protocol.

Standards define the vocabulary; IHE defines how to use that vocabulary in a concrete clinical workflow. Without integration profiles, two systems that “speak DICOM” may still be unable to work together — the workflow agreement is missing.

This chapter covers the IHE profiles most relevant to clinical information systems: the Scheduled Workflow (SWF) as the canonical radiology workflow model, XDS.b for cross-enterprise document sharing, PIX and PDQ for patient identity management, ATNA for security and audit, and the mobile profiles that bring those mechanisms to the FHIR ecosystem.

10.1 From Standard to Integration Profile

10.1.1 The problem of a standard without a profile

A standard such as HL7 v2.x defines the structure of an examination request message — but does not define who sends it, to whom, under what conditions, or in what sequence relative to other messages. Each vendor implements the standard in their own way; the result is systems that speak the same language but cannot understand each other in practice.

The scenario “with SWF” is testable, auditable and replaceable: any system that implements the SWF actors and transactions can be integrated without manual reconfiguration.

10.1.2 The IHE logic

IHE starts from a real clinical problem and defines a profile that addresses it using only existing standards. IHE does not invent protocols: it selects, combines and specifies. The process follows four steps:

  1. Identify the clinical problem — a real workflow with real stakeholders.
  2. Define the Actors — the systems that participate in the workflow, with their specific roles.
  3. Specify the Transactions — the numbered messages the Actors exchange, using existing standards (HL7, DICOM, FHIR).
  4. Publish the Profile — the complete script, with test cases for the Connectathon.

10.2 The IHE Triad: Actor, Transaction, Profile

The IHE vocabulary rests on three fundamental concepts: Actor, Transaction and Integration Profile. Understanding these three concepts is a prerequisite for reading any IHE Technical Framework document.

10.2.1 Actor

An Actor is a system or system component with a well-defined role in an IHE profile. An Actor is not a product — it is a role that a product may implement. The same product may implement multiple Actors from multiple profiles, declaring each combination in its Integration Statement.

The separation between Actor and product is deliberate: it allows the same RIS to implement both the Order Filler and Report Creator roles simultaneously, or for a product to be replaced by another without disrupting the rest of the workflow, provided the new product implements the same Actors.

10.2.2 Transaction

A Transaction is a numbered, normative message exchange between two Actors. Each Transaction has:

  • A code identifier (e.g. RAD-1, ITI-18, ITI-41)
  • An initiating Actor and a responding Actor
  • A base standard (HL7 v2.x, DICOM, FHIR REST)
  • A precise semantics — what the message contains, what it triggers, and what is expected in response

Transaction codes are unique within a Technical Framework domain. Transaction RAD-1 (Placer Order Management) is always the same — any vendor implementing the Order Placer Actor of SWF must support RAD-1 with the same semantics. Each transaction is specified down to the byte level: which HL7 field, which DICOM tag, which expected return code.

10.2.3 Integration Profile

An Integration Profile is a set of Actors and Transactions that addresses a specific clinical problem. The Profile specifies:

  • Which Actors exist and which are mandatory vs. optional
  • Which Transactions are exchanged between which Actors, and in what order
  • Which options and extensions are permitted
  • Which test cases must be passed at the Connectathon

A system declaring conformance with an IHE Profile guarantees — verified by independent testing — that it implements all the mandatory Actors and Transactions of that Profile.

10.3.2 Conformance and Connectathon

Conformance with an IHE Profile is not self-declared — it is tested at annual events called Connectathons. At a Connectathon:

  1. Vendors install their systems in a controlled network environment.
  2. Each system declares the Actors and Profiles it implements.
  3. An automated testing engine — Gazelle — executes the test cases defined in the Technical Framework and records the results.
  4. Vendors whose systems pass the tests receive IHE Certified status.

The results are publicly documented at gazelle.ihe.net: hospitals can check whether system X and system Y tested transaction T together — and with what outcome — before procurement decisions.

Note

Note — IHE conformance is not regulatory certification (CE marking or FDA). It is a technical interoperability commitment between vendors that complements — but does not replace — medical device certifications. IHE conformance means “we tested together at the Connectathon and it worked.”

10.4.3 End-to-end example — Ana Ferreira

Ana Ferreira, 46 years old, presents at CHUPorto Emergency with acute chest pain. The physician orders a chest CT with contrast. The SWF workflow proceeds as follows:

Step 1 — HIS → RIS (RAD-1): SAM registers the order: “Chest CT with contrast, urgent”. An HL7 ORM message is automatically sent to the Radiology RIS. Accession number generated: CHU-2026-TC-00847.

Step 2 — RIS → Worklist (RAD-4 / RAD-5): The RIS schedules the exam on CT-1 (Siemens SOMATOM Definition). The worklist is updated: "Ferreira, Ana · 1978-04-12 · Chest CT · CHU-2026-TC-00847". The technician opens the worklist on the CT console — sees the name and confirms. No manual entry.

Step 3 — CT acquires and stores (RAD-6 + RAD-8): MPPS “In Progress” sent to PACS at the start of acquisition. 320 images acquired in 3 minutes with correct DICOM metadata. Images sent via C-STORE. MPPS “Completed” sent at the end. PACS confirms receipt and indexes by Patient/Study/Series/Instance.

Step 4 — Radiologist reads (RAD-9): Automatic notification on the radiologist workstation. The study appears in the worklist. The radiologist opens the images, dictates and validates the report.

Step 5 — Report issued (RAD-27): HL7 ORU message sent to RIS. Report visible in SAM within less than 1 hour of the start of acquisition. From order to report with zero manual data entry.

10.5 Document Sharing — XDS.b

The XDS.b (Cross-Enterprise Document Sharing) profile addresses a different problem from SWF: not the intra-hospital flow of images, but the sharing of clinical documents between different institutions. A discharge summary, a pathology report, or an electronic prescription can be shared via XDS.b by any institution participating in an Affinity Domain — a trust network that has agreed to use the same central repository.

Consider the example from the slides: the GP at Gaia Health Centre wants to access the CT report produced at CHUPorto.

10.5.1 The four XDS.b Actors

The separation between Repository and Registry is a deliberate architectural decision: the Registry contains only metadata — document type, date, patient identifier, source institution — enabling fast searches without transferring documents. The Repository stores the actual documents and is only consulted after the Consumer has identified the document it requires. XDS.b separates storage (Repository) from the index (Registry) — each hospital keeps its own data; the Registry knows where to find it.

10.5.2 The XDS.b flow in three steps

Step 1 — Publication (Document Source → Repository and Registry):

CHUPorto’s PACS/RIS (Document Source) publishes the CT report. It sends the document to the Repository (ITI-41: Provide and Register Document Set) and registers the metadata in the Registry (ITI-42: Register Document Set).

ITI-41: document (CDA or FHIR Document) → Document Repository
ITI-42: metadata (type, date, patient ID, confidentiality) → Document Registry

Step 2 — Discovery (Document Consumer → Registry):

The GP at Gaia Health Centre queries the Registry to discover available documents for the patient (ITI-18: Registry Stored Query). The Registry returns a list of metadata entries with unique identifiers — the document itself is not transferred at this step.

ITI-18: Registry Stored Query
  patientId=urn:oid:1.2.840.10008&typeCode=18748-4  ← LOINC: Diagnostic imaging study
→ List of DocumentEntry with EntryUUID

Step 3 — Retrieval (Document Consumer → Repository):

The Consumer retrieves the document by its EntryUUID obtained in Step 2 (ITI-43: Retrieve Document Set). The Repository returns the complete document.

ITI-43: Retrieve Document Set
  documentUniqueId=<EntryUUID>
→ Complete CDA or FHIR Document

10.6 Patient Identity — PIX and PDQ

Patient identity is one of the hardest problems in healthcare interoperability. The same patient has different identifiers in each system: a record number at CHUPorto, another at CHUSJ, the national health number, the tax number (NIF). Without a correlation mechanism, any attempt to share data between systems risks associating records with the wrong patient.

Ana Ferreira has different IDs at CHUPorto (CHU-001) and CHUSJ (CHUSJ-456). How does the CHUSJ physician know they are the same patient?

10.6.1 PIX — Patient Identifier Cross-Referencing

The PIX profile defines a central component — the PIX Manager — that maintains a cross-reference table between identifiers for the same patient across different domains.

The PIX flow:

Registration: When a patient is admitted at CHUPorto, the HIS sends their local identifier (CHU-001) to the PIX Manager (ITI-8). When the same patient appears at CHUSJ as CHUSJ-456, the CHUSJ HIS also notifies the PIX Manager. The table is fed by admission feeds from each system.

Query: When CHUSJ needs to know who CHU-001 is in their system, it queries the PIX Manager (ITI-9):

PIX Query: "Who is CHU-001?"
PIX responds: "CHUSJ-456, NIF 123456789"
Instant resolution with no data duplication.

10.6.2 PDQ — Patient Demographics Query

PDQ addresses a different problem: the patient’s identifier is unknown — only the name and date of birth are available. PDQ allows the patient to be searched by demographic data in a central server:

PDQ Query: "Is there a patient: family=Ferreira, given=Ana, birthDate=1978-04-12?"
→ Returns a list of candidates with their identifiers.

If multiple results are returned, the user confirms the correct patient from the demographic data displayed (address, NIF, date of birth).

Note

Note — PIX resolves “who is this ID in another system”. PDQ resolves “does this patient exist and what is their ID”. In production environments, the two profiles are used together: PDQ locates the patient when the identifier is unknown; PIX confirms and cross-references when the identifier exists but belongs to a different domain.

10.7 Security and Audit — ATNA

The ATNA (Audit Trail and Node Authentication) profile defines the security mechanisms that are cross-cutting to all other IHE profiles. Any system participating in an XDS.b Affinity Domain or SWF deployment must implement ATNA. ATNA is not optional in production — it is a legal requirement under the GDPR and is required by all IHE profiles involving health data.

10.7.1 Secure Node — mutual TLS authentication

ATNA requires that all communication between IHE Actors be encrypted with mutual TLS (Transport Layer Security): both the client and the server present and validate an X.509 certificate. Any system — PACS, HIS, workstation — that accesses clinical data is required to use mutual TLS for all connections.

This is distinct from the unilateral TLS used in standard HTTPS: in ATNA, both ends authenticate each other. Auditable events include login, resource access, export, print and failure.

10.7.2 Audit Repository

ATNA requires each Actor to log an audit entry after each relevant event. The format is the DICOM Audit Message Schema (RFC 3881 / DICOM Sup 95); entries are sent to a central Audit Repository via Syslog over TLS.

Each entry records: who (user), what (resource), when (timestamp), where (network node), and why (event type).

Example: "Dr. Costa · Patient/CHU-001 · 14:32 · CT-Workstation-01 · read".

10.7.3 ATNA and the GDPR

ATNA is the technical implementation of the GDPR requirements in DICOM/HL7/FHIR environments. However, it addresses technical measures only — GDPR obligations around consent (Art. 6–9), data minimisation and international transfers (Chapter V) require organisational and legal measures beyond ATNA’s scope.

MHD transposes XDS.b to FHIR REST, preserving the same four-Actor architecture but replacing SOAP/ebXML with native FHIR operations.

In MHD, the document is published as a FHIR Bundle containing a DocumentReference (metadata), a Binary (the document — PDF, CDA or FHIR Document), and a List (manifest). The DocumentReference resource replaces the ebXML metadata of the classic XDS.b Document Entry.

10.8.3 IUA — Internet User Authorization

IUA defines OAuth2/SMART on FHIR authorisation for cross-institutional FHIR resource access. It defines scopes such as patient/*.read and user/Observation.write. The authorisation server issues a JWT access token; each FHIR request carries it in the Authorization: Bearer <token> header.

10.9 Integrated Scenario — Ana Ferreira, CHUPorto to CHUSJ

An urgent chest CT at CHUPorto. The report is accessed by the GP at CHUSJ three days later. Each step uses an IHE profile. One clinical episode — six IHE profiles working together, transparently to the clinician.

10.9.1 SWF — Order to PACS

HIS (SAM) → RIS → CT → PACS. DICOM worklist sent, C-STORE executed, MPPS confirmed. Images with correct metadata stored in CHUPorto’s PACS.

10.9.2 SWF — Report to HIS

The radiologist validates the report. HL7 ORU sent to HIS. A FHIR ImagingStudy resource is created by the PACS, referencing the DICOM images without moving them.

10.9.3 XDS.b — Report published to regional network

CHUPorto (Document Source) publishes the report (CDA or FHIR Document) to the regional Repository. A DocumentReference is registered in the regional Registry with metadata including patient ID, document type (18748-4 — CT report), date and confidentiality level.

10.9.4 PIXm — Identity reconciled

CHUSJ queries PIXm: “Who is CHU-001 in my system?” Answer: CHUSJ-456. The CHUSJ physician sees the correct patient in their local system without any manual reconciliation.

10.9.5 MHD — GP retrieves the report

GET /DocumentReference
    ?patient=Patient/CHUSJ-456
    &type=18748-4

The Registry returns the document reference. The GP downloads and reads the CHUPorto CT report directly from their FHIR-native system.

10.9.6 ATNA — All accesses audited

Every access — publication, query, download — generates an audit entry in the Audit Repository. GDPR Art. 30 and 32 obligations are met automatically.

10.10 Synthesis — FHIR and IHE as Complementary Layers

FHIR and IHE are not alternatives: they are complementary layers of the same interoperability stack. FHIR defines the vocabulary and the transport; IHE defines the scripts.

DICOM and HL7 define the vocabulary. IHE defines how to use that vocabulary in a concrete workflow. The “m” profiles bring REST/FHIR to IHE architecture. XDS, PIX and ATNA remain active — MHD, PDQm and IUA complement them. A modern system can be FHIR-native and IHE-compliant simultaneously.
