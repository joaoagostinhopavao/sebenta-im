6  HL7 v2 ŌĆö Message Model and Limitations

6.1 HL7 v2 and the Message Model

6.1.1 The problem HL7 v2 solved

Before HL7, each hospital information system operated as an isolated silo. Exchanging clinical data between systems ŌĆö for example, between the admissions system and the laboratory ŌĆö required manual intervention or proprietary point-to-point solutions. HL7 v2 was designed to solve this problem: an application-layer protocol (OSI layer 7) for the automatic exchange of clinical messages between heterogeneous systems (HL7 International, 2014).

HL7 v2 achieves a degree of semantic interoperability: data flows automatically between different systems with shared meaning. The unit of exchange is a message, sent when a clinical event occurs. The format is structured text, human-readable: segments separated by special characters (pipes and carets). The standard is open, maintained by the HL7 International consortium since 1987 (HL7 International, 2014).

The adoption was remarkable: more than 35 billion HL7 v2 messages are estimated to be processed every year worldwide, and the protocol is present in 95% of hospitals (Bhagat, 2019). Version 3 (2005) and FHIR (2011) coexist with v2 ŌĆö they have not replaced it.

6.1.2 A standard with 35 years of longevity

HL7 v2 is deeply embedded in existing hospital systems (HIS, LIS, RIS, PACS). Replacing it would require migrating decades of active interfaces. Table 6.1 summarises the main versions.

The longevity is not accidental: HL7 v2 aligns with real clinical workflows, is human-readable text, and benefits from a mature ecosystem of tools and consultants. Any clinical integration project in a hospital setting directly involves HL7 v2 messages in production (Benson & Grieve, 2021).

6.1.3 The trigger-event model

In HL7 v2, messages are not sent continuously ŌĆö they are created when something happens in the real world. A trigger event is a clinical occurrence that causes an HL7 message to be generated. Without an event there is no message: the system always waits for something to happen before communicating. The content of the message depends on the event that triggered it (Bhagat, 2019).

6.1.4 Message types and trigger codes

Each HL7 message is identified by two codes joined by a caret (^): the message type and the trigger event. The type is a three-letter uppercase acronym (e.g. ADT); the trigger is a letter followed by two digits (e.g. A01). Table 6.2 lists the most relevant types in a hospital context.

ADT^A01 means: Patient Administration (type) + Admit (trigger). The two codes together uniquely identify the message in any conformant implementation.

6.2 Structure of an HL7 v2 Message

6.2.1 Five-level hierarchy

An HL7 v2 message is composed of five levels of granularity ŌĆö from the broadest to the most specific (HL7 International, 2014):

  1. Message ŌĆö the complete unit of exchange (e.g. ADT^A01)
  2. Segment ŌĆö a logical grouping of related fields (e.g. MSH, PID, PV1)
  3. Field ŌĆö a data unit within a segment (e.g. PID-5 = patient name)
  4. Component ŌĆö a subdivision of a field, separated by ^ (e.g. family name ^ given name)
  5. Subcomponent ŌĆö a subdivision of a component, separated by & (e.g. code & institution)

6.2.2 The delimiters

Five special characters define the grammar of HL7 v2. They are declared in fields MSH-1 and MSH-2 of every message, meaning a parser can verify their presence before processing any other field (HL7 International, 2014).

MSH-1 always contains the pipe ŌĆö it is the field separator itself. MSH-2 contains the other four in order ^~\&. These two fields are the first in any message and allow the receiver to determine the delimiters before beginning to parse.

6.2.3 Segments: the logical unit

A segment is a logical grouping of related fields. It begins with a three-letter uppercase identifier followed by a pipe. The position determines the meaning of the field ŌĆö PID-5 is always the patient name, regardless of the implementation. Table 6.4 describes the essential segments in an ADT context.

The presence notation follows a normalised convention: no brackets means mandatory; [ ] means optional; { } means repeatable; [{ }] means optional and repeatable.

6.2.4 Fields: positional and fill states

Within a segment, fields are positional: the position number defines the meaning. A parser counts the pipes to know which field it is in. Trailing empty fields may be omitted ŌĆö the parser infers their absence.

A field can be in three distinct states with different semantics (HL7 International, 2014):

  • Populated ŌĆö the sending system provides a value. Example: 19610615
  • Not Populated ŌĆö the field exists but is empty; the sender does not have the value. Example: ||
  • Null ŌĆö an explicit instruction for the receiver to delete the value it holds. Example: |""|

The distinction between Not Populated and Null is critical: confusing the two can cause silent data corruption in the receiving system. Not Populated means ŌĆ£I donŌĆÖt have the valueŌĆØ; Null means ŌĆ£delete the value you haveŌĆØ.

6.3 Essential Segments

6.3.1 MSH ŌĆö Message header

The MSH is always the first segment and defines the full context of the message. Table 6.5 describes the most relevant fields, using the example from the lectures:

MSH|^~\&|ADT1|ULSTMAD|SCLINICO|ULSTMAD|200707181123||ADT^A01^ADT_A01|MSG00001|P|2.8

MSH-10 (Message Control ID) is the field the receiver returns in the ACK message to confirm which message was received ŌĆö it is the correlation mechanism between request and response.

6.3.2 EVN ŌĆö Event type

The EVN segment records the event type and the relevant timestamps. It appears immediately after the MSH:

EVN|A01|200707181123||

The most relevant fields are: EVN-1 (trigger code, e.g. A01), EVN-2 (moment the event was recorded in the system) and EVN-5 (identifier of the operator who recorded the event). EVN-3 (planned date) and EVN-4 (event reason) are optional and frequently empty.

In a transfer message (ADT^A02), the difference between EVN-2 (recording timestamp) and EVN-6 (actual event time) documents the interval between the clinical occurrence and its administrative registration.

6.3.3 PID ŌĆö Patient identification

The PID segment uniquely identifies the patient. Dissecting the example from the lectures illustrates field positional semantics (HL7 International, 2014):

PID|1||PATID1234^^^ULSTMAD^MR~123456789^^^PORTUGAL^SS||BAPTISTA^JOAO^A^JUNIOR
||19610615|M|||45^RUADOCARMO^^VILAREAL^VILAREAL^5000-678|||92212345
||S||PATID12345001^2^M10^ADT1^AN^A|

Field PID-3 illustrates repetition with ~: two groups separated by tilde correspond to two distinct identifiers for the same patient ŌĆö the hospitalŌĆÖs internal ID and the Social Security number. Within each group, ^ separates the identifier components: value, check digit, verification code, assigning authority, type and authority name.

6.3.4 PV1 ŌĆö Patient visit

The PV1 segment describes the current visit: physical location, attending physician and clinical service. Field PV1-3 uses a three-component structure identifying the location: ward^room^bed.

PV1|1|I|CIR-10^64^01||||124340^SILVA^ANTONIO|||SUR||||ADM

PV1-2 indicates the visit type (I = inpatient; O = outpatient; E = emergency). PV1-7 identifies the attending physician as code^family name^given name. PV1-10 indicates the clinical service (SUR = surgery, ICU = intensive care unit). In a discharge message, PV1-44 and PV1-45 record the admission date and discharge date respectively, allowing any receiving system to calculate the length of stay.

6.4 Message Confirmation: the ACK

After receiving a message, the destination system sends an ACK message to confirm receipt or to reject with error detail. The ACK always includes an MSA segment whose field MSA-2 repeats the MSH-10 of the original message, ensuring correlation between request and response (HL7 International, 2014).

The response codes in field MSA-1 are:

  • AA (Application Accept) ŌĆö message processed successfully
  • AE (Application Error) ŌĆö processed but with a content error
  • AR (Application Reject) ŌĆö rejected; processing was not possible

In a negative ACK (AE), the ERR segment provides error detail: field ERR-2 identifies the segment and field position with the error (PID^5 = field 5 of PID, patient name); ERR-3 contains the error code; ERR-4 indicates severity (E = error, W = warning). This mechanism allows the sender to diagnose exactly what failed.

6.5 ADT Messages: the PatientŌĆÖs Administrative Journey

ADT (Admission, Discharge, Transfer) messages track each administrative state change of the patient. Whenever a patient is admitted, changes service, or is discharged, an ADT message is generated and distributed to all subscribed systems. They are the most frequent messages in any hospital (HL7 International, 2014).

To illustrate the three central events, the lectures use a complete clinical case: Jo├Żo A. Baptista J├║nior, admitted to ULSTMAD on 18 July 2007 for surgery, transferred to ICU early the following morning, and discharged on 25 July.

The A01 event signals that a patient has been admitted and assigned a bed. It is the start of the hospital process: the HIS generates the ADT^A01 and distributes it to all subscribed systems (HL7 International, 2014).

6.6 HL7 v2 Limitations

6.6.1 Structural fragility: no formal schema

HL7 v2 has no formal schema. Any string is accepted in any field ŌĆö the standard does not define automatic format validation. Field PID-7 (date of birth) should contain YYYYMMDD, but the receiver must accept whatever arrives. Format errors pass silently until they surface in a wrong clinical report (HL7 International, 2014).

PID|1||PAT001||SILVA^ANA||19850312|F|...    ŌåÉ YYYYMMDD ŌĆö correct
PID|1||PAT001||SILVA^ANA||1985-03-12|F|...  ŌåÉ accepted, but different format
PID|1||PAT001||SILVA^ANA||12/03/1985|F|...  ŌåÉ accepted, yet another dialect

A database with a DATE column automatically rejects 15/06/1961. In HL7 v2, no such mechanism exists. Each sending system has its own dialect for dates, genders and codes; the integration engine accumulates normalisation rules per system. As the hospital network grows, this normalisation work becomes a permanent ongoing cost.

6.6.2 Weak semantics: local value tables

HL7 v2 defines fields by name and position, but does not define the vocabulary of their values. The specification provides user-defined tables: each hospital chooses its own codes. What should be a shared vocabulary becomes a private dialect of each institution (Benson & Grieve, 2021).

The most common example is PV1-4 (admission type):

All three hospitals are conformant with the specification ŌĆö and are incompatible with each other. The same problem exists in PID-8 (gender), clinical service, diagnosis type and marital status: every field with a user-defined table is a potential point of incompatibility. Two ŌĆ£HL7 v2 conformantŌĆØ systems may have completely different semantics and require bilateral mapping before any data exchange.

6.6.3 Z-segments: private extensions

Any segment whose identifier begins with Z is reserved for local use. The specification prohibits Z-segments in the normalised standard ŌĆö they are intentionally private and opaque. Each hospital creates its own extra fields (e.g. ZPI for insurance information, ZIN for health subsystem data) without publishing them (HL7 International, 2014).

The problem becomes apparent when integrating two hospitals:

Hospital A:  ZIN|ADSE|GROUP001|PLANO_B
Hospital B:  ZSG|ADSE|PLANO_B|GROUP001

The same content, different identifiers, different field order ŌĆö mutually incompatible. An external parser does not know what ZIN or ZSG mean; it must be configured case by case, by bilateral agreement. Z-segments are the most visible example of a broader problem: the standard is open enough to generate incompatibilities even between conformant implementations.

6.6.4 Parsing: hidden complexity

Processing an HL7 v2 message is more demanding than simply splitting strings on the pipe character. Three main sources of complexity can be identified (Bhagat, 2019):

  • Repeatable fields with ~ ŌĆö a field may have multiple values separated by tilde; PID-3 can contain several identifiers for the same patient; the parser must handle lists, not single values.
  • Opaque Z-segments ŌĆö a generic parser does not know what ZIN, ZPI or ZCU mean; it must be configured case by case.
  • Version differences ŌĆö the same field may have different structures between v2.3 and v2.8; in a real environment, different systems send different versions and the parser must handle all of them.

There is no universal parser for HL7 v2: every environment requires system-specific and version-specific rules.

6.6.5 The integration engine: the required piece

The limitations described above converge on one point: any real HL7 v2 environment requires an integration middleware ŌĆö commonly called an integration engine. Products such as Mirth Connect, Rhapsody or InterSystems Ensemble receive messages from multiple systems, normalise formats, map local tables, translate Z-segments and route to the correct destinations (Benson & Grieve, 2021).

The integration engine is not optional: it is the piece that makes HL7 v2 work in heterogeneous environments. The cost the standard does not show is the maintenance cost of this middleware: each new system joining the network requires analysis, mapping and testing. As N systems grow, the number of interfaces ŌĆö and the cost of maintaining them ŌĆö grows with it. This is the structural limit of the bilateral model that HL7 v2 imposes.

6.6.6 Balance: what HL7 v2 solved and where it fell short

HL7 v2 solved a real problem: it eliminated the manual exchange of data between hospital systems, created a trigger-event model aligned with clinical workflows, and survived 35 years of technological change. Its limits only became apparent as scale grew.

### Message types and trigger codes

Each HL7 message is identified by two codes joined by a caret (`^`): the
message type and the trigger event. The type is a three-letter uppercase
acronym (e.g. `ADT`); the trigger is a letter followed by two digits
(e.g. `A01`). @tbl-msg-types lists the most relevant types in a hospital
context.

| Type | Full name | Trigger | Clinical event |
|------|-----------|---------|----------------|
| ADT | Patient Administration | A01 | Patient admission |
| ADT | Patient Administration | A02 | Patient transfer |
| ADT | Patient Administration | A03 | Patient discharge |
| ADT | Patient Administration | A04 | Outpatient / emergency registration |
| ORM | Order Message | O01 | Lab or exam order |
| ORU | Observation Result | R01 | Lab result |
| MDM | Medical Document Management | T02 | Clinical document creation |
| DFT | Detail Financial Transaction | P03 | Financial / billing transaction |

: Message types and main triggers in HL7 v2 [@HL7_v28; @Bhagat2019]. {#tbl-msg-types}

`ADT^A01` means: *Patient Administration* (type) + *Admit* (trigger).
The two codes together uniquely identify the message in any conformant
implementation.

## Structure of an HL7 v2 Message {#sec-hl7v2-structure}

### Five-level hierarchy

An HL7 v2 message is composed of five levels of granularity ŌĆö from the
broadest to the most specific [@HL7_v28]:

1. **Message** ŌĆö the complete unit of exchange (e.g. `ADT^A01`)
2. **Segment** ŌĆö a logical grouping of related fields (e.g. `MSH`, `PID`, `PV1`)
3. **Field** ŌĆö a data unit within a segment (e.g. `PID-5` = patient name)
4. **Component** ŌĆö a subdivision of a field, separated by `^` (e.g. family name `^` given name)
5. **Subcomponent** ŌĆö a subdivision of a component, separated by `&` (e.g. code `&` institution)

### The delimiters

Five special characters define the grammar of HL7 v2. They are declared
in fields `MSH-1` and `MSH-2` of every message, meaning a parser can
verify their presence before processing any other field [@HL7_v28].

| Character | Name | Function | Example |
|-----------|------|----------|---------|
| `&#124;` | Pipe | Field separator | `PID&#124;1&#124;&#124;PAT001&#124;&#124;SILVA^ANA` |
| `^` | Caret | Component separator | `SILVA^ANA^M` ŌåÆ family name ^ given name ^ initial |
| `~` | Tilde | Repetition separator | `PID-3: ID001~ID002` ŌåÆ two alternative IDs |
| `\` | Backslash | Escape character | `\F\` represents a literal pipe in content |
| `&` | Ampersand | Subcomponent separator | `ID001&UTAD` ŌåÆ ID ^ issuing institution |

: HL7 v2 delimiters ŌĆö declared in MSH-1 and MSH-2 [@HL7_v28]. {#tbl-delimiters}

`MSH-1` always contains the pipe ŌĆö it is the field separator itself.
`MSH-2` contains the other four in order `^~\&`. These two fields are
the first in any message and allow the receiver to determine the
delimiters before beginning to parse.

### Segments: the logical unit

A **segment** is a logical grouping of related fields. It begins with a
three-letter uppercase identifier followed by a pipe. The position
determines the meaning of the field ŌĆö `PID-5` is always the patient name,
regardless of the implementation. @tbl-segments describes the essential
segments in an ADT context.

| ID | Name | Presence | Function |
|----|------|----------|---------|
| MSH | Message Header | Mandatory, unique | Sender, receiver, message type, version |
| EVN | Event Type | Mandatory | Event type and occurrence timestamp |
| PID | Patient Identification | Mandatory | Name, ID, date of birth, gender, address |
| PV1 | Patient Visit | Mand./Optional | Ward, bed, attending physician, service |
| NK1 | Next of Kin | Optional, repeatable | Family members or emergency contacts |
| NTE | Notes and Comments | Optional, repeatable | Free-text notes |
| ERR | Error | Optional | Error detail in negative ACK messages |

: Essential segments in ADT messages [@HL7_v28]. {#tbl-segments}

The presence notation follows a normalised convention: no brackets means
mandatory; `[ ]` means optional; `{ }` means repeatable; `[{ }]` means
optional and repeatable.

### Fields: positional and fill states

Within a segment, fields are **positional**: the position number defines
the meaning. A parser counts the pipes to know which field it is in.
Trailing empty fields may be omitted ŌĆö the parser infers their absence.

A field can be in three distinct states with different semantics
[@HL7_v28]:

- **Populated** ŌĆö the sending system provides a value. Example: `19610615`
- **Not Populated** ŌĆö the field exists but is empty; the sender does not
  have the value. Example: `||`
- **Null** ŌĆö an explicit instruction for the receiver to delete the value
  it holds. Example: `|""|`

The distinction between *Not Populated* and *Null* is critical: confusing
the two can cause silent data corruption in the receiving system. *Not
Populated* means "I don't have the value"; *Null* means "delete the value
you have".

## Essential Segments {#sec-hl7v2-segments}

### MSH ŌĆö Message header

The MSH is always the first segment and defines the full context of the
message. @tbl-msh describes the most relevant fields, using the example
from the lectures:

```
MSH|^~\&|ADT1|ULSTMAD|SCLINICO|ULSTMAD|200707181123||ADT^A01^ADT_A01|MSG00001|P|2.8
```

| Field | Value | Name | Meaning |
|-------|-------|------|---------|
| MSH-1 | `&#124;` | Field Separator | The pipe itself ŌĆö the only field whose value is the delimiter |
| MSH-2 | `^~\&` | Encoding Characters | The four secondary delimiters |
| MSH-3 | `ADT1` | Sending Application | Sending application |
| MSH-4 | `ULSTMAD` | Sending Facility | Sending institution |
| MSH-5 | `SCLINICO` | Receiving Application | Receiving application |
| MSH-6 | `ULSTMAD` | Receiving Facility | Receiving institution |
| MSH-7 | `200707181123` | Date/Time | Timestamp: YYYYMMDDHHII |
| MSH-9 | `ADT^A01^ADT_A01` | Message Type | Type ^ Trigger ^ structure |
| MSH-10 | `MSG00001` | Message Control ID | Unique ID ŌĆö returned by receiver in ACK |
| MSH-11 | `P` | Processing ID | P = production ┬Ę T = test ┬Ę D = debug |
| MSH-12 | `2.8` | Version ID | HL7 version used |

: Main fields of the MSH segment [@HL7_v28]. {#tbl-msh}

`MSH-10` (*Message Control ID*) is the field the receiver returns in
the ACK message to confirm which message was received ŌĆö it is the
correlation mechanism between request and response.

### EVN ŌĆö Event type

The EVN segment records the event type and the relevant timestamps.
It appears immediately after the MSH:

```
EVN|A01|200707181123||
```

The most relevant fields are: `EVN-1` (trigger code, e.g. `A01`),
`EVN-2` (moment the event was recorded in the system) and `EVN-5`
(identifier of the operator who recorded the event). `EVN-3` (planned
date) and `EVN-4` (event reason) are optional and frequently empty.

In a transfer message (`ADT^A02`), the difference between `EVN-2`
(recording timestamp) and `EVN-6` (actual event time) documents the
interval between the clinical occurrence and its administrative
registration.

### PID ŌĆö Patient identification

The PID segment uniquely identifies the patient. Dissecting the example
from the lectures illustrates field positional semantics [@HL7_v28]:

```
PID|1||PATID1234^^^ULSTMAD^MR~123456789^^^PORTUGAL^SS||BAPTISTA^JOAO^A^JUNIOR
||19610615|M|||45^RUADOCARMO^^VILAREAL^VILAREAL^5000-678|||92212345
||S||PATID12345001^2^M10^ADT1^AN^A|
```

| Field | Value | Meaning |
|-------|-------|---------|
| PID-1 | `1` | Set ID ŌĆö segment order number |
| PID-3 | `PATID1234^^^ULSTMAD^MR~123456789^^^PORTUGAL^SS` | Two IDs: internal MR + Social Security number |
| PID-5 | `BAPTISTA^JOAO^A^JUNIOR` | Name: family name ^ given name ^ initial ^ suffix |
| PID-7 | `19610615` | Date of birth ŌĆö 15 Jun 1961 |
| PID-8 | `M` | Gender ŌĆö M (Male) |
| PID-11 | `45^RUADOCARMO^^VILAREAL^...^5000-678` | Structured address: street ^ ^ city ^ ŌĆ” ^ postal code |

: Main fields of the PID segment, Jo├Żo Baptista example [@HL7_v28]. {#tbl-pid}

Field `PID-3` illustrates repetition with `~`: two groups separated by
tilde correspond to two distinct identifiers for the same patient ŌĆö the
hospital's internal ID and the Social Security number. Within each group,
`^` separates the identifier components: value, check digit, verification
code, assigning authority, type and authority name.

### PV1 ŌĆö Patient visit

The PV1 segment describes the current visit: physical location,
attending physician and clinical service. Field `PV1-3` uses a
three-component structure identifying the location:
`ward^room^bed`.

```
PV1|1|I|CIR-10^64^01||||124340^SILVA^ANTONIO|||SUR||||ADM
```

`PV1-2` indicates the visit type (`I` = inpatient; `O` = outpatient;
`E` = emergency). `PV1-7` identifies the attending physician as
`code^family name^given name`. `PV1-10` indicates the clinical service
(`SUR` = surgery, `ICU` = intensive care unit). In a discharge message,
`PV1-44` and `PV1-45` record the admission date and discharge date
respectively, allowing any receiving system to calculate the length of
stay.

## Message Confirmation: the ACK {#sec-hl7v2-ack}

message to confirm receipt or to reject with error detail. The ACK
always includes an `MSA` segment whose field `MSA-2` repeats the
`MSH-10` of the original message, ensuring correlation between request
and response [@HL7_v28].

The response codes in field `MSA-1` are:

- **AA** (*Application Accept*) ŌĆö message processed successfully
- **AE** (*Application Error*) ŌĆö processed but with a content error
- **AR** (*Application Reject*) ŌĆö rejected; processing was not possible

## ADT Messages: the Patient's Administrative Journey {#sec-hl7v2-adt}

**ADT** (*Admission, Discharge, Transfer*) messages track each
administrative state change of the patient. Whenever a patient is
admitted, changes service, or is discharged, an ADT message is generated
and distributed to all subscribed systems. They are the most frequent
messages in any hospital [@HL7_v28].

To illustrate the three central events, the lectures use a complete
clinical case: Jo├Żo A. Baptista J├║nior, admitted to ULSTMAD on 18 July
2007 for surgery, transferred to ICU early the following morning, and
discharged on 25 July.

### ADT^A01 ŌĆö Admission

The `A01` event signals that a patient has been admitted and assigned
a bed. It is the start of the hospital process: the HIS generates the
`ADT^A01` and distributes it to all subscribed systems [@HL7_v28].

In the case study, on 18 July 2007 at 11:23, Jo├Żo Baptista is admitted
to the surgery service (SUR), Ward CIR-10, Room 64, Bed 01, with
Dr. Ant├│nio Silva (#124340) as attending physician. The complete
message is:


The message has five segments: MSH (envelope), EVN (event timestamp),
PID (patient identification, three lines), NK1 (spouse Nilza F. Pereira)
and PV1 (location and physician). `PV1-3` uses the structure
`CIR-10^64^01` ŌĆö ward ^ room ^ bed.

The `A02` event is generated when an admitted patient changes location

In the early hours of 19 July, Jo├Żo Baptista's condition worsens due

location (`UCI^01^02`), `PV1-6` = previous location (`CIR-10^64^01`),
`PV1-8` = Dr. Ferreira (new responsible), `PV1-10` = ICU service.

The `A03` event records the end of the hospital stay. Field `PV1-36`

## HL7 v2 Limitations {#sec-hl7v2-limitations}

### Structural fragility: no formal schema

HL7 v2 has no formal schema. Any string is accepted in any field ŌĆö the
standard does not define automatic format validation. Field `PID-7`
(date of birth) should contain `YYYYMMDD`, but the receiver must accept
whatever arrives. Format errors pass silently until they surface in a
wrong clinical report [@HL7_v28].

```
PID|1||PAT001||SILVA^ANA||19850312|F|...    ŌåÉ YYYYMMDD ŌĆö correct
PID|1||PAT001||SILVA^ANA||1985-03-12|F|...  ŌåÉ accepted, but different format
PID|1||PAT001||SILVA^ANA||12/03/1985|F|...  ŌåÉ accepted, yet another dialect
```

A database with a `DATE` column automatically rejects `15/06/1961`. In
HL7 v2, no such mechanism exists. Each sending system has its own dialect
for dates, genders and codes; the integration engine accumulates
normalisation rules per system. As the hospital network grows, this
normalisation work becomes a permanent ongoing cost.

### Weak semantics: local value tables

HL7 v2 defines fields by name and position, but does not define the
vocabulary of their values. The specification provides **user-defined
tables**: each hospital chooses its own codes. What should be a shared
vocabulary becomes a private dialect of each institution
[@BensonGrieve2021].

The most common example is `PV1-4` (admission type):

| Hospital | Value in PV1-4 | Local meaning |
|----------|----------------|---------------|
| Hospital A | `U` | Urg├¬ncia (Emergency) |
| Hospital B | `E` | Emergency |
| Hospital C | `1` | Urgent |

: Weak semantics: the same field, incompatible values across hospitals [@BensonGrieve2021]. {#tbl-semantics}

All three hospitals are conformant with the specification ŌĆö and are
incompatible with each other. The same problem exists in `PID-8`
(gender), clinical service, diagnosis type and marital status: every
field with a *user-defined table* is a potential point of
incompatibility. Two "HL7 v2 conformant" systems may have completely
different semantics and require bilateral mapping before any data
exchange.

### Z-segments: private extensions

Any segment whose identifier begins with `Z` is reserved for local use.
The specification prohibits Z-segments in the normalised standard ŌĆö they
are intentionally private and opaque. Each hospital creates its own
extra fields (e.g. `ZPI` for insurance information, `ZIN` for health
subsystem data) without publishing them [@HL7_v28].

The problem becomes apparent when integrating two hospitals:

```
Hospital A:  ZIN|ADSE|GROUP001|PLANO_B
Hospital B:  ZSG|ADSE|PLANO_B|GROUP001
```

The same content, different identifiers, different field order ŌĆö
mutually incompatible. An external parser does not know what `ZIN` or
`ZSG` mean; it must be configured case by case, by bilateral agreement.
Z-segments are the most visible example of a broader problem: the
standard is open enough to generate incompatibilities even between
conformant implementations.

### Parsing: hidden complexity

Processing an HL7 v2 message is more demanding than simply splitting
strings on the pipe character. Three main sources of complexity can be
identified [@Bhagat2019]:

- **Repeatable fields with `~`** ŌĆö a field may have multiple values
  separated by tilde; `PID-3` can contain several identifiers for the
  same patient; the parser must handle lists, not single values.
- **Opaque Z-segments** ŌĆö a generic parser does not know what `ZIN`,
  `ZPI` or `ZCU` mean; it must be configured case by case.
- **Version differences** ŌĆö the same field may have different structures
  between v2.3 and v2.8; in a real environment, different systems send
  different versions and the parser must handle all of them.

There is no universal parser for HL7 v2: every environment requires
system-specific and version-specific rules.

### The integration engine: the required piece

The limitations described above converge on one point: any real HL7 v2
environment requires an **integration middleware** ŌĆö commonly called an
*integration engine*. Products such as Mirth Connect, Rhapsody or
InterSystems Ensemble receive messages from multiple systems, normalise
formats, map local tables, translate Z-segments and route to the correct
destinations [@BensonGrieve2021].

The integration engine is not optional: it is the piece that makes HL7
v2 work in heterogeneous environments. The cost the standard does not
show is the maintenance cost of this middleware: each new system joining
the network requires analysis, mapping and testing. As N systems grow,
the number of interfaces ŌĆö and the cost of maintaining them ŌĆö grows with
it. This is the structural limit of the bilateral model that HL7 v2
imposes.

### Balance: what HL7 v2 solved and where it fell short

HL7 v2 solved a real problem: it eliminated the manual exchange of data
between hospital systems, created a trigger-event model aligned with
clinical workflows, and survived 35 years of technological change. Its
limits only became apparent as scale grew.

| Aspect | Assessment |
|--------|------------|
| Elimination of manual data exchange | Solved |
| Trigger-event model aligned with clinical workflows | Solved |
| Global adoption (95% of hospitals) | Solved |
| Mature ecosystem of tools | Solved |
| Formal schema with automatic validation | Not solved |
| Shared semantics across institutions | Not solved |
| Interoperable private extensions (Z-segments) | Not solved |
| Universal parser | Not solved |
| Integration cost proportional to number of systems | Not solved |

## Summary and Exercises {#sec-hl7v2-summary}

### Consolidation exercises {.unnumbered}

1. A laboratory system (LIS) receives an order sent by the prescribing
   system. Identify: (a) the HL7 message type used; (b) the trigger
   event; (c) the MSH fields that allow the LIS to determine who sent
   the order and which HL7 version it conforms to.

2. The `PID-3` field of a message contains the value
   `PATID1234^^^ULSTMAD^MR~123456789^^^PORTUGAL^SS`.
   Explain the role of the `~` character in this context and identify
   the two identifiers present, stating the type of each.

3. A negative ACK (AE) contains `ERR||PID^5|101^Required field missing|E`.
   Interpret each component of this segment and explain the practical
   consequence for the sending system.

4. Two hospitals use different values in `PV1-4` to classify emergency
   admissions: Hospital X uses `U`, Hospital Y uses `E`. Both claim
   to be HL7 v2 conformant. Explain how this situation is possible and
   what architectural component is needed to integrate them.

5. A hospital needs to transmit health insurance data (subsystem code,
   group number, plan) between the HIS and the pharmacy. The HL7 v2
   specification defines no standard segment for this purpose. What
   mechanism does the standard provide? What are the implications for
   interoperability with other hospitals?
