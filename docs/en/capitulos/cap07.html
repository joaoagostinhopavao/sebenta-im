7  Web Architecture and REST

7.1 From Message to Resource — Evolution of Communication Models in Healthcare

7.1.1 The message model and its limitations

HL7 v2 solved a fundamental problem: it eliminated the manual exchange of data between hospital information systems and created an automatic model based on clinical messages. However, as hospital networks grew and inter-institutional interoperability became necessary, the structural limits of the model became evident (Benson & Grieve, 2021).

The central problem is tight coupling: in a message-based model, each pair of systems must be configured individually. If a hospital has N systems (HIS, LIS, RIS, PACS, pharmacy, etc.), the number of interfaces to configure and maintain grows on the order of N², since each system must have a specific interface for every other. With ten systems, there are potentially ninety bilateral interfaces. Each new interface requires analysis, development, testing and ongoing maintenance. Scalability is structurally limited.

Alongside this, the message model is push-based: information is transmitted when an event occurs, not when it is needed. An external system that needs to query the current state of a patient cannot simply ask — it must wait for the relevant event to generate a message and deliver it. Ad hoc querying is not part of the model.

7.1.2 From SOAP/XML to REST/JSON

Before REST established itself as the dominant paradigm, integration between more recent healthcare systems relied on Web Services SOAP — Simple Object Access Protocol — a communication protocol based on XML that structures messages in a rigid format called an envelope: a structure with header and body that wraps the data to be transmitted. HL7 v3, released in 2005, was designed precisely to use this model: voluminous XML messages transported in SOAP envelopes, consumed by Web Services servers, managed by specialised middleware (Benson & Grieve, 2021). Middleware — intermediary software that mediates communication between systems — was mandatory in HL7 v3 architectures. In practice, it required specialised integration engines such as Mirth Connect or Rhapsody, responsible for receiving, transforming and routing SOAP messages.

Complexity was its greatest obstacle. A typical SOAP request required: generating the XML envelope, configuring the Web Services server, installing and maintaining integration middleware (of the ESB — Enterprise Service Bus type), and dealing with a data model (RIM — Reference Information Model) too abstract for direct implementation. In many institutions, this implementation cost seriously limited the adoption of HL7 v3 (Benson & Grieve, 2021).

REST proposes a radically simpler model: any HTTP server can serve clinical resources. It need not be a specific server (that is, a dedicated special webservice). A request is purely HTTP — a GET /Patient/123 — and the response is JSON. There is no SOAP envelope, no mandatory middleware. Any web developer can implement a REST server without knowledge of hospital middleware — the barrier to entry dropped drastically.

7.1.3 The web as universal infrastructure

HTTP — HyperText Transfer Protocol — has existed since 1991. It is the most widely deployed application protocol in the world: supported by every programming language, operating system and device (R. Fielding et al., 2022). Its adoption as the transport for clinical interoperability was not accidental — it represents the deliberate choice of a universally available infrastructure over a proprietary stack.

This choice has direct practical consequences:

  • Any hospital with a web server can expose clinical data via REST
  • The development and debugging tools are the same used in any web project
  • Security (HTTPS, token-based authentication) is shared with the global web ecosystem
  • Maintenance draws on a vast knowledge base of web developers

It is this alignment with the universal infrastructure of the web that explains why FHIR, the modern clinical interoperability standard, adopted REST as its native communication model.

7.2 The REST Model — Principles, Constraints and Stateless

7.2.1 What REST is

REST — Representational State Transfer — is an architectural style for distributed systems, proposed by Roy Fielding in his doctoral dissertation in 2000 (R. T. Fielding, 2000). It is not a protocol, not a formal standard, not a data format. It is a set of constraints that, when respected, produce systems with specific properties: scalability, simplicity and interoperability.

The distinction matters: a system can use HTTP and JSON without being REST; and it can be REST without using JSON. What qualifies a system as RESTful (conformant with REST) is adherence to the architectural constraints defined by Fielding (R. T. Fielding, 2000).

7.2.2 The six architectural constraints

Fielding defined six constraints. A system that satisfies all of them is said to be RESTful:

1. Client–server — Separation of concerns: the client manages the interface and user experience; the server manages data and business logic. The two evolve independently. A FHIR server can change its internal implementation without requiring clients to be updated — provided the REST interface remains stable.

2. Stateless — The server retains no session state between requests. Each client request contains all the information necessary to be processed independently: resource identity, authentication credentials, search parameters. This constraint is the most critical for scalability — see the following section.

3. Cacheable — Responses must indicate whether they may be stored in cache. Resources that change infrequently (such as terminologies or lists of specialties) can be cached by the client, reducing server load and response latency.

4. Uniform interface — The interface between client and server is standardised and predictable, regardless of the resource type. It rests on four sub-constraints: identification of resources by URI — Uniform Resource Identifier; manipulation of resources through their representations; self-descriptive messages (each message contains sufficient information to be interpreted without external context); and HATEOAS — Hypermedia as the Engine of Application State — the server includes in its responses links to the next possible states.

5. Layered system — The client need not know whether it communicates directly with the final server or with intermediaries (proxies, load balancers, API gateways). Layers are transparent to the client. This property enables horizontal scaling and the addition of security without altering the interface.

6. Code on demand (optional) — The server may send executable code to the client (for example, JavaScript). It is the only optional constraint. In clinical contexts, this capability is rarely used; most REST implementations in healthcare ignore this constraint without ceasing to be considered RESTful.

7.2.3 Stateless — the central principle

Of all the constraints, stateless is the one with the greatest practical impact. Understanding it in depth is essential to appreciating why REST scales where other models fail.

In a stateful model, the server maintains a session per client: it knows who is connected, what was previously requested, language preferences, and the current point in navigation. The client need only send the differences relative to the previous state. This reduces the volume of data transmitted — but creates a rigid dependency between client and server. If the server fails, the session is lost. Horizontal scaling (adding servers) requires session synchronisation across instances (R. T. Fielding, 2000).

In a stateless model, each request is completely autonomous. The client always sends all required information: the resource identity, credentials, parameters. The server processes and responds with no memory of previous requests. If the server fails, the client can resend exactly the same request to a different server with no loss of context. Any server instance can respond to any request — horizontal scalability is immediate (R. T. Fielding, 2000; Richardson & Ruby, 2007).

The most accessible illustration of stateless is a web search: each request to a search engine includes the terms, language filters and location. The server has no knowledge of what was searched before — there is no session to maintain. The same request can be processed by any server instance in the data centre. It is precisely this model that REST generalises to any type of resource — including clinical resources.

Note

Note — The term stateless refers to session state on the server, not to the state of data. A REST server has state in its data (resources exist and are persisted); what it lacks is memory of the conversation with the client. Data is maintained — the session is not.

7.3 CRUD and HTTP Methods

7.3.1 The four fundamental operations

CRUD — Create, Read, Update, Delete — describes the four operations possible on any information resource. These are primitive and complete operations: any data manipulation can be expressed as a combination of the four. In REST, each CRUD operation has a corresponding HTTP method, which creates a universal vocabulary shared by any implementation (R. Fielding et al., 2022; Richardson & Ruby, 2007).

The property of idempotency deserves attention: a method is idempotent when executing it N times produces the same result as executing it once. GET, PUT and DELETE are idempotent — repeating the request has no additional effect. POST is not idempotent: sending the same POST twice creates two distinct resources. This distinction is relevant in clinical environments where network reliability cannot be guaranteed — a client that does not know whether its request was received can safely repeat a GET or a DELETE, but must not repeat a POST without first verifying whether the resource has already been created (R. Fielding et al., 2022).

7.3.2 Structure of an HTTP request and response

All HTTP communication follows a fixed cycle: the client sends a request and the server returns a response. Both have a normalised structure (R. Fielding et al., 2022).

A request is composed of three parts:

  1. Request line — HTTP method (see table Table 7.2) + URL + HTTP version. Example: GET /api/courses/42 HTTP/1.1.
  2. Headers — metadata: destination server, accepted content type, authentication credentials. Plain text, inspectable with any tool (curl, Postman, browser). The Host header is mandatory; the rest are optional.
  3. Body — data sent (used in POST and PUT; absent in GET and DELETE).

A response is composed of:

  1. Status line — HTTP version + status code + message. Example: HTTP/1.1 200 OK
  2. Headers — response content type, size, location of the created resource (in POST).
  3. Body — the requested resource, typically in JSON.
GET /api/utentes/P-12345 HTTP/1.1
Host: fhir.snssul.min-saude.pt
Accept: application/json
Authorization: Bearer eyJhbGci...
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 412

{
  "id": "P-12345",
  "nome": "Ana Silva",
  "dataNascimento": "1978-04-22",
  "nrUtente": "123456789"
}

7.4 Endpoints and URLs — Structure and Search Parameters

7.4.1 What an endpoint is

An endpoint is a specific address that a REST server exposes to allow interaction with a resource type. It is identified by a URL — Uniform Resource Locator — and an HTTP method.

The combination of method + URL uniquely defines the operation: GET /Patient/123 and POST /Patient are two different endpoints, even though they share the same base path.

7.4.2 Anatomy of a REST URL

A well-formed REST URL follows a predictable hierarchical structure (Berners-Lee et al., 2005):

https://fhir.snssul.min-saude.pt/api/v1/Patient/P-12345?_format=json
│       │                            │   │        │      │
│       │                            │   │        │      └─ search parameter
│       │                            │   │        └───────── resource id
│       │                            │   └────────────────── resource type
│       │                            └────────────────────── base path (API version)
│       └─────────────────────────────────────────────────── server
└─────────────────────────────────────────────────────────── protocol

Each component has a defined role:

  • Protocol (https://) — HTTPS ensures the communication is encrypted. HTTP is acceptable in local development; never in production with clinical data.
  • Server — domain name of the server hosting the API. Different subdomains may host distinct APIs: fhir.hospital.pt for clinical data, pacs.hospital.pt for imaging.
  • Base path — prefix identifying the API and, frequently, its version (/api/v1). Allows the API to evolve without breaking existing clients: a client using /v1 continues to function when /v2 is released.
  • Resource type — name of the collection, by convention in singular or plural. In FHIR: Patient, Observation, DiagnosticReport, Appointment.
  • Resource ID — unique identifier of the resource within this collection. If omitted (GET /Patient), the request returns the collection (list of all resources, typically with pagination).
  • Search parameters — after ?, filters and modifiers. See section 7.6.

7.4.3 Naming conventions

Naming conventions for REST endpoints have evolved into a set of widely adopted practices (Richardson & Ruby, 2007):

  • Nouns, not verbs — /Patient and not /getPatient or /createPatient; the verb is the HTTP method
  • Hierarchy for relationships — /Patient/123/Observation for the observations of patient 123
  • Opaque IDs — the client must not infer semantics from the ID; /Patient/abc-123 is preferable to /Patient/2026-silva-ana
  • Version in the base path — /api/v1/Patient allows the API to evolve without breaking clients

7.5 JSON — Structure, Data Types and Readability

7.5.1 What JSON is

JSON — JavaScript Object Notation — is the most widely used data serialisation format in web APIs. Defined by the ECMA-404 standard (Ecma International, 2017), it is plain text structured as key–value pairs, human-readable and directly processable by any modern programming language.

Despite the name including JavaScript, JSON is completely independent of any language. In Python, Java, C#, R or any other language, native functions exist for serialising and deserialising JSON without external libraries.

7.5.2 Fundamental syntax

JSON syntax is defined by six rules:

  1. Object — a collection of key–value pairs delimited by curly braces { }. Keys are always strings in double quotes. Pairs are separated by commas.
  2. Array — an ordered sequence of values delimited by square brackets [ ]. Elements are separated by commas.
  3. String — a sequence of characters in double quotes. Example: "Ana Silva".
  4. Number — an integer or decimal number, without quotes. Example: 42, 3.14.
  5. Boolean — true or false, without quotes.
  6. Null — an explicit null value: null.

Any value may be an object, array, string, number, boolean or null — and objects and arrays may be nested to any depth.

7.5.3 Data types in JSON

{
  "id": "OBS-001",
  "codigo": "8480-6",
  "designacao": "Tensão arterial sistólica",
  "valor": 122,
  "unidade": "mmHg",
  "dataHora": "2025-11-14T09:30:00+00:00",
  "critico": false,
  "componentes": [
    { "codigo": "8480-6", "valor": 122, "designacao": "Sistólica" },
    { "codigo": "8462-4", "valor": 78,  "designacao": "Diastólica" }
  ],
  "notas": null
}

In this example:

  • "id", "codigo", "designacao", "unidade", "dataHora" — strings (in double quotes)
  • "valor" → 122 — integer number, without quotes
  • "critico" → false — boolean
  • "componentes" → [...] — array of objects (two elements)
  • "notas" → null — explicit null value (the field exists but has no value)
Note

Note — In JSON, the difference between "122" (string) and 122 (number) is semantic, not merely a matter of format. A system that expects a number and receives a string may fail or behave incorrectly. Dates and times are always strings in JSON (there is no native date type) — the normalised representation is ISO 8601: "2025-11-14T09:30:00+00:00".

7.5.4 JSON vs XML in a clinical context

XML — Extensible Markup Language — was the dominant format in the APIs of the previous generation (Web Services SOAP, HL7 v3 CDA). A direct comparison illustrates the practical differences:

  <unidade>mmHg</unidade>

7.5.5 A complete clinical resource in JSON

The following example represents a patient conforming to the FHIR model, in JSON, to illustrate how a real clinical resource is structured:

{
  "resourceType": "Patient",
  "id": "P-12345",
  "meta": {
    "versionId": "3",
    "lastUpdated": "2025-11-14T08:00:00+00:00"
  },
  "identifier": [
    {
      "system": "https://www.sns.gov.pt/utente",
      "value": "123456789"
    }
  ],
  "name": [
    {
      "use": "official",
      "family": "Silva",
      "given": ["Ana", "Maria"]
    }
  ],
  "gender": "female",
  "birthDate": "1978-04-22",
  "address": [
    {
      "line": ["Rua do Comércio, 45"],
      "city": "Vila Real",
      "postalCode": "5000-678",
      "country": "PT"
    }
  ],
  "telecom": [
    { "system": "phone", "value": "+351 259 123 456" }
  ]
}

Each field has a precise role:

  • "resourceType" — identifies the FHIR resource type; always present
  • "id" — the system ID: a technical identifier assigned by the server; it is what appears in the URL /Patient/P-12345
  • "meta" — resource metadata: version (versionId) and last update (lastUpdated)
  • "identifier" — array of clinical identifiers: the NHS patient number; may contain several (record number, NIF, etc.)
  • "name", "address", "telecom" — arrays because a person may have multiple names, addresses and contacts

Note the distinction between "id" (system ID) and "identifier" (clinical ID). The "id" is technical and internal to the server — it may change if data are migrated to another system. The NHS patient number in "identifier" is the real-world identifier, independent of any technology, recognised and used by clinicians in day-to-day practice.

7.6 Search Parameters and Chaining — Composite Searches

7.6.1 Basic search parameters

In REST, resource searches are performed through search parameters (search parameters) appended to the URL after the ? symbol. Each parameter is a name=value pair. Multiple parameters are combined with &, forming a logical AND (Berners-Lee et al., 2005; Richardson & Ruby, 2007).

The general structure is:

GET /[ResourceType]?[parameter1]=[value1]&[parameter2]=[value2]

Progressive examples with clinical resources:

# Single parameter: observations for a patient
GET /Observation?patient=P-12345

# Two parameters (AND): observations for a patient within a period
GET /Observation?patient=P-12345&date=ge2025-01-01

# Three parameters: laboratory observations for a patient in 2025
GET /Observation?patient=P-12345&date=ge2025-01-01&category=laboratory

# Pagination: first 10 results, sorted by date
GET /Observation?patient=P-12345&_count=10&_sort=-date

# Laboratory reports between two dates
GET /DiagnosticReport?patient=P-12345
  &date=ge2025-01-01
  &date=le2025-12-31
  &category=LAB

# Patients whose family name contains "silva"
GET /Patient?family:contains=silva

# Observations with systolic blood pressure above 140
GET /Observation?code=8480-6&value-quantity=gt140

7.6.3 Chaining — chained searches

Chaining is an advanced search capability that allows filtering a resource type based on attributes of a referenced resource — without requiring multiple sequential HTTP requests (Richardson & Ruby, 2007).

The notation uses a full stop (.) as the separator between the reference and the attribute of the referenced resource:

GET /[ResourceType]?[reference-field].[referenced-attribute]=[value]

To illustrate the concept, consider a clinical scenario: retrieve all observations whose patient has a particular NHS number.

Without chaining (two sequential requests):

# Step 1: find the patient ID by NHS number
GET /Patient?identifier=123456789
→ returns { "id": "P-12345", ... }

# Step 2: use the obtained ID to search for observations
GET /Observation?patient=P-12345

With chaining (a single request):

GET /Observation?patient.identifier=123456789

The server resolves the reference internally: it traverses the observations, navigates to the patient referenced in each one, checks the identifier, and returns only the observations whose patient holds that NHS number.

7.6.4 Chaining examples in a hospital context

# Observations for female patients
GET /Observation?patient.gender=female

# DiagnosticReports for patients born after 1990
GET /DiagnosticReport?patient.birthdate=gt1990-01-01

# Observations performed by a clinician from a specific department
GET /Observation?performer.organization.name=Laboratório+Central

# Double chaining: observations for patients with address in Vila Real
GET /Observation?patient.address-city=Vila+Real

Chaining can be nested across multiple levels (separated by additional full stops), although the supported depth varies across implementations. The principle remains: each full stop traverses a reference between resources.

Note

Note — Chaining significantly increases search expressiveness, but carries a performance cost: the server must resolve references internally, which may entail multiple database operations. In production FHIR servers, deep chaining queries are frequently subject to timeout or complexity limits. When implementing clients, simple chaining (one level) should be preferred wherever possible.
