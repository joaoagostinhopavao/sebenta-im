4  DICOM as a Data Model

4.1 The DICOM Standard

4.1.1 The problem before DICOM

Until the early 1980s, digital medical imaging was produced by equipment whose storage format was defined exclusively by each manufacturer. A General Electric computed tomography scanner could not export images to a Siemens workstation; a Philips magnetic resonance system stored its files in a format incompatible with any other system. This fragmentation created an absolute dependency on the vendor — the so-called vendor lock-in — and made it impossible to share images between institutions, even when such sharing would have been clinically relevant (Pianykh, 2012).

The practical consequences were significant: radiological films circulated physically in cardboard envelopes, with risk of loss and deterioration; comparison with previous examinations depended on the availability of the physical archive; and integration with hospital information systems was non-existent. A paradox thus arose: acquisition equipment was technologically advanced, yet the information it produced remained in a proprietary, isolated format (Huang, 2010).

4.1.2 Definition, scope, and governance

The DICOM standard (Digital Imaging and Communications in Medicine) is the international standard for digital medical imaging. Published as ISO 12052:2017 and managed by MITA (Medical Imaging & Technology Alliance), a division of NEMA (National Electrical Manufacturers Association), the standard is continuously updated — there are no fixed major versions, but dated editions; the current edition is designated PS3 2025e (NEMA / MITA, 2025e) (as of April 2026).

The standard is organised into 22 active parts, each covering a specific domain: from introduction and conformance definition (PS3.1–PS3.2) to data structure (PS3.5–PS3.6), message services (PS3.7), network protocol (PS3.8), file format (PS3.10), and security profiles (PS3.15). More than thirty working groups contribute to its ongoing maintenance and evolution.

In terms of scope, the standard covers all relevant clinical imaging modalities: computed tomography (CT), magnetic resonance (MR), ultrasonography (US), digital radiography (DX), nuclear medicine (NM), radiotherapy, ophthalmology, and others. The standard’s coverage is therefore transversal to all digital medical imaging (International Organization for Standardization, 2017; NEMA / MITA, 2025e).

4.1.3 Functional capabilities and ecosystem

The DICOM standard covers the complete lifecycle of medical imaging: from producing — creating images in a standardised format on any equipment; storing — preserving images with all associated metadata; transmitting — sending between devices and institutions; querying — finding studies by patient, date, or modality; displaying — presenting images on diagnostic workstations with appropriate parameters; to printing — sending to film or paper printers (NEMA / MITA, 2025e).

From an ecosystem perspective, the standard benefits all stakeholders: clinicians gain rapid access to images and reports, including via remote diagnosis; patients benefit from faster care and data that can be transferred between institutions; hospitals can purchase equipment from different manufacturers with guaranteed compatibility; and the industry has access to a broad, competitive market, provided its products comply with the standard (Pianykh, 2012).

4.2 The DICOM Data Model

4.2.1 From the real world to the DICOM world

The conceptual starting point of the DICOM standard is the mapping between entities in the real world (in a clinical context) and their formal, standardised representation in computer systems. In the real world, a patient has a name, identifier, date of birth, sex, and weight; in the standard, each of these attributes is encoded through a standardised numeric code (TAG), a well-defined data type, and a value that follows precise formatting rules (NEMA / MITA, 2025d).

This mapping is ensured by a dictionary and guarantees that, regardless of the equipment manufacturer, storage system, or visualisation workstation, the same attribute is always represented in the same way. The concept “patient name” is always encoded by the same dictionary entry, in the same format; “date of birth” always follows the YYYYMMDD notation; “sex” accepts only the enumerated values M, F, or O. Uniformity is not a recommendation — it is an obligation imposed by the standard.

4.2.2 Data Elements, TAGs, and the Dictionary (PS3.6)

The fundamental unit of information in the DICOM standard is the Data Element. In the file — the so-called wire format (PS3.5) — each Data Element is encoded with exactly four fields: the TAG, the VR (Value Representation), the Value Length, and the Value Field (NEMA / MITA, 2025c). These four fields are everything that exists in the byte-stream transmitted or stored; there are no other fields in the file.

The TAG is a pair of 16-bit hexadecimal integers written in the form (gggg,eeee), where gggg is the group number and eeee the element number. The group number thematically groups attributes: group 0010 contains all patient attributes, group 0008 the modality identification attributes, group 0020 the study attributes, group 0028 the image parameters, and group 7FE0 contains the pixel data itself (NEMA / MITA, 2025d).

The Data Dictionary, defined in part PS3.6 of the standard, constitutes the official registry of all normalised Data Elements. Each dictionary entry is a six-field tuple — but note: this is a different level from the Data Element in the file. Of the six fields, only three (TAG, Name, and VR) are encoded in the wire format (PS3.5); the remaining three (VM, Type, and Description) belong exclusively to the dictionary model — they define the semantic contract of the attribute but do not appear as fields in the file (NEMA / MITA, 2025c, 2025d). The Type field indicates the requirement level: 1 = mandatory and non-empty; 1C = conditional; 2 = mandatory but may be empty; 3 = optional. This distinction is relevant for validating the conformance of DICOM systems.

The VM (Value Multiplicity) field indicates how many values are permitted in the Value Field of a given attribute. VM 1 means a single value; VM 1-n means one or more values; VM 2-2n means an even number of values. For example, the attribute (0008,0061) Modalities in Study has VM 1-n — it may contain a single modality (CT) or several (CT, MR, US), depending on the number of series of different types present in the study. Note that VM is a restriction declared in the dictionary; the Value Field in the file does not explicitly carry the number of values — the receiver infers it from the length (Value Length) and the type (VR) (NEMA / MITA, 2025c, 2025d).

4.2.3 Value Representations (VR)

Value Representations (VR) define the data type of each Data Element. The standard specifies 27 VR types, constituting the DICOM equivalent of data types in a programming language — int, string, date, among others (NEMA / MITA, 2025c).

4.2.4 Integrated view: TAG + VR + VM

The three concepts — TAG, VR, and VM — articulate in a precise relationship: the TAG identifies what, the VR defines how the value is encoded, and the VM indicates how many values are permitted in the Value Field. This triple contract is declared in the Data Dictionary for each normalised attribute (NEMA / MITA, 2025d).

4.3 Entities, IODs, and Modules

4.3.1 The Patient → Study → Series → Image hierarchy

The DICOM information model organises data in a four-level hierarchy that reflects the real clinical workflow. Each level corresponds to an Information Entity (IE) and contains one or more elements of the immediately lower level (NEMA / MITA, 2025b).

Patient — represents the person. Stores demographic data: name, identifier, date of birth, sex. A patient may have multiple studies over time.

Study — corresponds to a clinical imaging episode: a date, a description, a referring physician, and a globally unique study UID. Each study contains one or more series.

Series — corresponds to a specific acquisition within the study. It identifies the modality (CT, MR…), the series number, and the patient’s position. A brain CT study may have, for example, five series: non-contrast, contrast-enhanced, bone reconstruction, 3D reconstruction, and localiser image (scout).

Image — is the atomic unit of information: an individual image, with its pixel dimensions, bit depth, compression parameters, and above all, the Pixel Data itself. Each image is identified by a globally unique, non-repeating UID generated by the acquisition equipment at capture time.

The practical consequence of this hierarchy is direct: a single multi-series brain CT can generate approximately 1,500 distinct DICOM files — each containing all the metadata from the four levels (Patient, Study, Series, Image) plus the pixels of that specific image (Huang, 2010).

4.3.2 Information Object Definitions (IOD)

An IOD (Information Object Definition) is the formal specification of the set of attributes that describes a particular type of DICOM object. It is not a file — it is a normative definition: it states which attributes are mandatory, conditional, and optional for a given type of medical image or document (NEMA / MITA, 2025b).

The standard defines a composite IOD (Composite IOD) for each relevant modality: CT Image IOD, MR Image IOD, CR Image IOD, US Image IOD, and others. A fundamental point that deserves explicit attention: a composite IOD always describes a single image — not a study or a series. The IOD is instantiated once per image; a CT with 300 slices generates 300 instances of the CT Image IOD, each in an independent DICOM file.

Each instance carries with it the attributes of all hierarchical levels: patient, study, series, and image data. The logical grouping into series and studies is not performed by physical references between files — it is inferred from shared UIDs: all images in the same series share the same Series Instance UID; all images in the same study share the same Study Instance UID. The hierarchy exists in the metadata of each individual file; any system reading them can reconstruct it without requiring external structures (NEMA / MITA, 2025b).

Each composite IOD is derived from a general entity model (DICOM Composite Instance IOD Information Model) that formalises the relationships between Patient, Study, Equipment, Frame of Reference, Series, and the various types of possible instances (Image, SR Document, Waveform, Surface, etc.) (NEMA / MITA, 2025b).

For example, the CR Image IOD (Computed Radiography) includes, in terms of information entities (IE): Patient, Study, Series, Equipment, and Image — all present in each individual file. Each IE is specified by the modules that belong to it, with their respective usage level.

4.3.3 Information modules and usage types (M/C/U)

Each IE of an IOD is composed of modules — thematic groupings of related Data Elements. A module always has an associated usage level (NEMA / MITA, 2025b):

  • M (Mandatory) — mandatory; must be present and populated.
  • C (Conditional) — conditional; mandatory only when a specific condition is met (e.g., the Contrast/Bolus module is C — mandatory if contrast medium was used).
  • U (User Optional) — optional; may be included or omitted by the manufacturer.

The Patient Module (reference C.7.1.1), mandatory (M) in all composite IODs, groups the patient identification attributes: (0010,0010) Patient's Name (Type 2), (0010,0020) Patient ID (Type 2), (0010,0030) Patient's Birth Date (Type 3), and (0010,0040) Patient's Sex (Type 2), among others. Note that Type 2 means the attribute must be present but may have an empty value — something clinically relevant in emergency contexts (NEMA / MITA, 2025b).

4.3.4 The DICOM file: storage format

The composite IOD defines which attributes describe an image; part PS3.10 of the standard defines how those attributes are physically stored in a file. These are two distinct planes: the model and the packaging.

A DICOM file begins with a preamble of 128 bytes (reserved, usually null) followed by the ASCII prefix DICM. Next comes the File Meta Information — a set of group 0002 Data Elements describing the file itself: the format version, the stored SOP Class, the instance UID, and the negotiated Transfer Syntax (which determines the compression format of the Pixel Data). After this header follows the sequence of IOD Data Elements, ordered by ascending TAG (NEMA / MITA, 2025a).

The Transfer Syntax is an important detail: it tells the receiver whether the Pixel Data is in raw format (uncompressed), JPEG Baseline, JPEG 2000, or RLE. This information is in the file header — not in the Pixel Data itself — which allows the receiver to select the correct decoder before reading the pixels.

A representative excerpt from a CT DICOM file, as displayed in a metadata viewer such as dcmdump (DCMTK), illustrates the sequence of Data Elements ordered by TAG:

(0008,0060) CS  Modality              [CT]
(0008,1030) LO  Study Description     [BRAIN CT]
(0010,0010) PN  Patient's Name        [COUTO^FERNANDO]
(0010,0020) LO  Patient ID            [987654]
(0010,0030) DA  Patient's Birth Date  [19650412]
(0028,0010) US  Rows                  [512]
(0028,0011) US  Columns               [512]
(7FE0,0010) OW  Pixel Data            [binary — 524288 bytes]

Note that all metadata from the four hierarchical levels (Patient, Study, Series, Image) are present in each individual DICOM file. This intentional redundancy ensures that any file is self-sufficient and can be interpreted without depending on an external database (NEMA / MITA, 2025a; Pianykh, 2012).

4.4 Summary and Exercises

Consolidation exercises

  1. A CT DICOM file contains the Data Element (0010,0030) DA 8 19820915. Interpret each field (TAG, VR, Value Length, Value Field), indicating the thematic group to which the TAG belongs and the clinical meaning of the value.

  2. Explain why a composite CT Image IOD with 300 images generates 300 independent DICOM files rather than a single file. How is the Study/Series hierarchy reconstructed from those files?

  3. Distinguish the fields that physically exist in a serialised Data Element from the fields that belong exclusively to the corresponding entry in the Data Dictionary. Why is this distinction relevant for implementing a DICOM system?

  4. A Data Dictionary entry has VM 1-n and Type 2. What do these two values mean together for a system validating the conformance of a received DICOM file?

  5. The 128-byte preamble of a DICOM file is corrupted. Discuss the impact of this corruption on the ability of a PS3.10-conformant system to process the file.
