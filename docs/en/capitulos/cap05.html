5  DICOM Services and PACS Architecture

5.1 DICOM Services — DIMSE: images in action

5.1.1 Application Entities, SCU, and SCP

Beyond the data model covered in the previous chapter, DICOM also defines how information is manipulated, through the definitions contained in DIMSE (DICOM Message Service Element).

DIMSE is the services layer of the standard, specified in part PS3.7 (NEMA / MITA, 2025d).

The first concept to consider in any communications system concerns the entities responsible for communication, i.e., sender and receiver. In a DICOM system, all communication occurs between Application Entities (AE). An AE is any device or application that implements the DICOM standard: a CT scanner, a PACS archive server, a visualisation workstation, or a printer. Each AE is uniquely identified by an AE Title — an alphanumeric string configured by the system administrator.

In each interaction, an AE assumes one of two roles:

  • SCU (Service Class User) — the entity that requests the service, sending the request (-Rq). This entity may also be informally identified as the client.
  • SCP (Service Class Provider) — the entity that provides the service, responding to the request (-Rsp). This entity may equally be informally identified as the server.

SCU and SCP roles are not fixed: the same AE may be SCU in one interaction (when sending images to the archive) and SCP in another (when receiving a query request from a workstation). This flexibility is fundamental to the operation of distributed systems with multiple devices (NEMA / MITA, 2025d).

DIMSE services follow a standardised naming convention: the prefix C- indicates that the service operates on composite data (Composite); the prefix N- indicates normalised data (Normalized). The suffix -Rq identifies the request and -Rsp the response.

5.1.2 Structure of a DIMSE message

Each DIMSE message is composed of two elements: a DICOM Command Object and, optionally, a DICOM Data Object. The Command Object is encoded as a set of group 0000 Data Elements — following exactly the same TAG + VR + Value structure as the other Data Elements, but defined in part PS3.7 (not in the Data Dictionary PS3.6). The most relevant command fields are (0000,0100) (Command Field, which identifies the service type), (0000,0110) (Message ID), and (0000,0800) (Data Set Type, which indicates whether an IOD is attached) (NEMA / MITA, 2025d).

The Data Set Type field has the value 0101h when no IOD is attached (as in C-ECHO) and another value when the IOD is present (as in C-STORE). This distinction allows the receiver to know, before processing the message body, whether or not to expect additional data. The complete structure of a DIMSE message is illustrated in Figure 5.1.

5.1.3 The four main services: C-ECHO, C-STORE, C-FIND, C-MOVE

C-ECHO — verifies the existence of a functional DICOM connection between two AEs. It goes beyond a simple TCP/IP ping: it confirms that the DICOM application layer is operational. It carries no IOD. It is routinely used to monitor the availability of system components (NEMA / MITA, 2025d).

C-STORE — sends a complete DICOM instance from one AE to another. It is the most widely used service in the clinical workflow: it occurs immediately after each acquisition, when the equipment sends images to the archive server. The equipment acts as SCU and the server as SCP.

C-FIND — queries studies, series, or images on a remote AE based on criteria such as patient identifier, date, modality, or study description. The SCU specifies the criteria; the SCP returns a list of matching results. Note that C-FIND does not transfer images — only metadata (NEMA / MITA, 2025d).

C-MOVE — requests the SCP to transfer images to a third AE. The SCU specifies the destination by AE Title; the SCP uses C-STORE to send the images to that destination. This indirect mechanism is typical of visualisation workstations, which ask the server to send images directly to themselves. Figure 5.2 illustrates the request–response model of the three main services.

5.1.4 SOP Class: the service–object contract

A SOP Class (Service-Object Pair Class) is the formal combination of a group of DIMSE services with an IOD. It defines, normatively, what can be done with a particular type of information object. Each SOP Class is identified by a globally unique UID, registered in part PS3.6 (NEMA / MITA, 2025b, 2025c).

The paradigmatic example is the CT Image Storage SOP Class: it combines the C-STORE service with the CT Image IOD, identified by the UID 1.2.840.10008.5.1.4.1.1.2. When a CT scanner sends images to an archive server, both devices negotiate during the connection establishment (Association Negotiation) whether they support this SOP Class — the transfer only occurs if both support it (NEMA / MITA, 2025b).

Note that an IOD is not necessarily an image with pixels: the Modality Worklist IOD contains the list of pending examinations for a modality, without any pixel data; and the Verification SOP Class (C-ECHO) has no associated IOD at all. 5.2 PACS Systems

5.2.1 From analogue radiology to PACS

A PACS (Picture Archiving and Communication System) is the computer system that manages the complete lifecycle of medical images in a hospital institution: from acquisition at the modalities through to archiving, distribution, and visualisation for diagnostic and reporting purposes (Huang, 2010).

The radiology workflow before PACS was entirely manual: patient registration with the printing of a paper request; examination performance and development of analogue film; manual search of the physical archive to locate previous examinations; transport of dossiers to the radiologist; dictation of the report; transcription; review and signing; and insertion of the final report into the RIS (Huang, 2010).

With PACS, this workflow is fully digitised: images arrive directly from the equipment to the archive via DICOM C-STORE, previous examinations are automatically pre-loaded onto diagnostic workstations, the radiologist works in an integrated digital environment, and the signed report is distributed electronically. The time between acquisition and diagnostic availability drops from hours to seconds.

Documented benefits include the elimination of analogue film costs (estimated at between 6 and 8% of the radiology budget in pre-PACS hospitals), the reduction of repeated examinations due to inability to access previous studies, and support for remote diagnosis (teleradiology) (Huang, 2010).

5.2.2 General architecture and workflow

In a generic PACS, the RIS (Radiology Information System) coordinates the administrative and clinical flow, whilst the DICOM protocol transports images between the technical components. The integrated workflow comprises:

  1. The patient registers in the RIS, which assigns a unique identifier.
  2. The RIS sends HL7 messages with examination data to the PACS Broker.
  3. The PACS Broker notifies the archive server of the appointment.
  4. Relevant previous examinations are pre-loaded onto diagnostic workstations (prefetch).
  5. The examination room queries the DICOM Modality Worklist (C-FIND on the Modality Worklist IOD) to obtain the pending examination data.
  6. The technician performs the examination; the equipment sends the images (C-STORE) to the quality control workstation (QC WS).
  7. The QC WS forwards the examination to the radiologist’s diagnostic workstation.
  8. The examination is archived in the PACS server (database updated).
  9. The server distributes the examination to clinical department workstations.
  10. The radiologist dictates and signs the report; the database is marked as complete.
  11. The report is transcribed and distributed electronically (Huang, 2010).

5.2.3 Architecture types: stand-alone, client-server, and web

Three fundamental PACS architectures can be identified, with different trade-offs between resilience, flexibility, and network dependency (Huang, 2010):

Stand-alone (thick client) — images are automatically distributed by the server to diagnostic workstations, which maintain local storage. The workstation can continue operating even if the server becomes unavailable. The disadvantage is that each workstation has its own worklist, creating a risk of duplicate reading of the same examination. The query/retrieve function (C-FIND + C-MOVE) allows non-pre-loaded images to be retrieved from the archive.

Client-server (thin client) — workstations have no local storage; they access the central worklist directly and load images from server memory. The worklist is single and shared, eliminating duplicate readings. The structural disadvantage is that the server constitutes a single point of failure (SPOF): any interruption prevents access to images.

Web-based — images are served via HTTPS through the DICOMweb interface (PS3.18), which exposes DICOM services as a REST API. Any device with a compatible browser can access the archive without installing specialised software. This model shares the network vulnerabilities of the client-server approach, but adds platform independence and natural support for teleradiology (Huang, 2010). Without MWL, the technician would manually enter the patient data into the equipment — with risk of transcription errors. With MWL, data flows automatically from the RIS to the equipment, ensuring that all generated DICOM files already contain the correct metadata from the very first moment (Huang, 2010).

5.3 Security in DICOM and PACS

5.3.1 Normative mechanisms: CMS, TLS, and HTTPS

The DICOM standard has defined security mechanisms since 1999, driven by HIPAA (Health Insurance Portability and Accountability Act). Specified in part PS3.15, they are organised in three layers (NEMA / MITA, 2025a):

CMS — Cryptographic Message Syntax — operates at the DICOM object level. It allows encryption of sensitive parts, notably patient data or the Pixel Data. The protection is permanent: it persists in the file regardless of the transport medium or destination system.

TLS — Transport Layer Security — operates at the TCP/IP connection level during transfer via DIMSE. It creates an encrypted channel between two AEs during the session. Protection exists only in transit: the object at the destination is unprotected after reception, unless CMS is also applied.

HTTPS — Web Services — when objects are served via DICOMweb (PS3.18, REST API over HTTP), the standard mandates the use of HTTPS for the transport layer. Like TLS, protection is restricted to the communication channel (NEMA / MITA, 2025a).

The three mechanisms are not mutually exclusive: a robust implementation combines CMS on objects with TLS on DIMSE connections and HTTPS on web interfaces, covering data at rest and in transit. Figure 5.4 illustrates the three layers and the scope of protection of each.

5.3.2 Implementation responsibility

A critical aspect of the DICOM standard: it defines the available mechanisms but does not impose them. The decision to apply security rests with the hospital institution and the system vendor (NEMA / MITA, 2025a).

In practice, many installations operate without active TLS on internal DIMSE connections — acceptable in a segmented, controlled network, but problematic with connections to external networks or in the face of unauthorised internal access risk. The minimum alternative is access exclusively via VPN; however, this approach does not protect objects at the destination.

In the context of the GDPR and obligations for health data (special category, Article 9), the absence of encryption on DICOM files containing identifying patient data constitutes a legal and organisational risk that institutions must actively assess and mitigate.

### SOP Class: the service–object contract

A **SOP Class** (*Service-Object Pair Class*) is the formal combination
of a group of DIMSE services with an IOD. It defines, normatively, what
can be done with a particular type of information object. Each SOP Class
is identified by a globally unique UID, registered in part PS3.6
[@DICOM_PS34; @DICOM_PS36].

The paradigmatic example is the **CT Image Storage SOP Class**: it
combines the C-STORE service with the CT Image IOD, identified by the
UID `1.2.840.10008.5.1.4.1.1.2`. When a CT scanner sends images to an
archive server, both devices negotiate during the connection establishment
(*Association Negotiation*) whether they support this SOP Class — the
transfer only occurs if both support it [@DICOM_PS34].

Note that an IOD is not necessarily an image with pixels: the
*Modality Worklist* IOD contains the list of pending examinations for a
modality, without any pixel data; and the *Verification* SOP Class
(C-ECHO) has no associated IOD at all. @tbl-sop presents representative
examples of normalised SOP Classes:

| SOP Class | IOD | Service |
|---|---|---|
| CT Image Storage `1.2.840.10008.5.1.4.1.1.2` | CT Image IOD | C-STORE |
| MR Image Storage `1.2.840.10008.5.1.4.1.1.4` | MR Image IOD | C-STORE |
| Study Root Q/R — FIND `1.2.840.10008.5.1.4.1.2.2.1` | Study Root IOD | C-FIND |
| Study Root Q/R — MOVE `1.2.840.10008.5.1.4.1.2.2.2` | Study Root IOD | C-MOVE |
| Modality Worklist `1.2.840.10008.5.1.4.31` | Modality Worklist IOD | C-FIND |
| Verification `1.2.840.10008.1.1` | *(no IOD)* | C-ECHO |

: Examples of normalised SOP Classes [@DICOM_PS34; @DICOM_PS36]. {#tbl-sop}

It can thus be seen that DIMSE, IOD, and SOP Class have distinct but
complementary roles: DIMSE defines the message format; the IOD defines
the data structure; and the SOP Class declares the valid contract between
both, identified by a negotiable UID.

## PACS Systems {#sec-pacs}

### From analogue radiology to PACS

A **PACS** (*Picture Archiving and Communication System*) is the
computer system that manages the complete lifecycle of medical images
in a hospital institution: from acquisition at the modalities through
to archiving, distribution, and visualisation for diagnostic and
reporting purposes [@Huang2010].

The radiology workflow before PACS was entirely manual: patient
registration with the printing of a paper request; examination
performance and development of analogue film; manual search of the
physical archive to locate previous examinations; transport of dossiers
to the radiologist; dictation of the report; transcription; review and
signing; and insertion of the final report into the RIS [@Huang2010].

With PACS, this workflow is fully digitised: images arrive directly
from the equipment to the archive via DICOM C-STORE, previous
examinations are automatically pre-loaded onto diagnostic workstations,
the radiologist works in an integrated digital environment, and the
signed report is distributed electronically. The time between acquisition
and diagnostic availability drops from hours to seconds.

Documented benefits include the elimination of analogue film costs
(estimated at between 6 and 8% of the radiology budget in pre-PACS
hospitals), the reduction of repeated examinations due to inability to
access previous studies, and support for remote diagnosis
(*teleradiology*) [@Huang2010].

### General architecture and workflow

In a generic PACS, the **RIS** (*Radiology Information System*)
coordinates the administrative and clinical flow, whilst the DICOM
protocol transports images between the technical components. The
integrated workflow comprises:

1. The patient registers in the RIS, which assigns a unique identifier.
2. The RIS sends HL7 messages with examination data to the **PACS Broker**.
3. The PACS Broker notifies the archive server of the appointment.
4. Relevant previous examinations are pre-loaded onto diagnostic
   workstations (*prefetch*).
5. The examination room queries the DICOM **Modality Worklist** (C-FIND
   on the *Modality Worklist IOD*) to obtain the pending examination data.
6. The technician performs the examination; the equipment sends the images
   (C-STORE) to the quality control workstation (QC WS).
7. The QC WS forwards the examination to the radiologist's diagnostic
   workstation.
8. The examination is archived in the PACS server (database updated).
9. The server distributes the examination to clinical department
   workstations.
10. The radiologist dictates and signs the report; the database is marked
    as complete.
11. The report is transcribed and distributed electronically [@Huang2010].

### Architecture types: stand-alone, client-server, and web

Three fundamental PACS architectures can be identified, with different
trade-offs between resilience, flexibility, and network dependency
[@Huang2010]:

**Stand-alone (thick client)** — images are automatically distributed
by the server to diagnostic workstations, which maintain local storage.
The workstation can continue operating even if the server becomes
unavailable. The disadvantage is that each workstation has its own
*worklist*, creating a risk of duplicate reading of the same examination.
The *query/retrieve* function (C-FIND + C-MOVE) allows non-pre-loaded
images to be retrieved from the archive.

**Client-server (thin client)** — workstations have no local storage;
they access the central *worklist* directly and load images from server
memory. The *worklist* is single and shared, eliminating duplicate
readings. The structural disadvantage is that the server constitutes a
single point of failure (SPOF): any interruption prevents access to
images.

**Web-based** — images are served via HTTPS through the **DICOMweb**
interface (PS3.18), which exposes DICOM services as a REST API. Any
device with a compatible browser can access the archive without
installing specialised software. This model shares the network
vulnerabilities of the client-server approach, but adds platform
independence and natural support for teleradiology [@Huang2010]. Figure
5.3 compares the stand-alone and client-server architectures.

### PACS integration with HIS and RIS

PACS integrates bidirectionally with the **HIS** (*Hospital Information
System*) and the **RIS** (*Radiology Information System*) through HL7
messages — a clinical interoperability standard addressed in a later
chapter. @tbl-integracao summarises the most relevant flows:

| From        | To          | Message / Service              | Protocol     |
|-------------|-------------|--------------------------------|--------------|
| HIS         | RIS         | Patient admission (ADT A01)    | HL7 v2       |
| RIS         | PACS Broker | Examination booking (ORM O01)  | HL7 v2       |
| RIS         | Modality    | Examination worklist (MWL)     | DICOM        |
| Modality    | PACS        | Acquired images                | DICOM C-STORE|
| PACS        | RIS         | Examination confirmation (ORU R01) | HL7 v2   |

: PACS–HIS–RIS integration flows [@Huang2010]. {#tbl-integracao}

The **Modality Worklist** (MWL) mechanism deserves particular attention:
it is through this mechanism that the acquisition equipment receives,
via C-FIND, the list of pending examinations. Without MWL, the
technician would manually enter the patient data into the equipment —
with risk of transcription errors. With MWL, data flows automatically
from the RIS to the equipment, ensuring that all generated DICOM files
already contain the correct metadata from the very first moment
[@Huang2010].

## Security in DICOM and PACS {#sec-dicom-seguranca}

### Normative mechanisms: CMS, TLS, and HTTPS

The DICOM standard has defined security mechanisms since 1999, driven
by HIPAA (*Health Insurance Portability and Accountability Act*).
Specified in part PS3.15, they are organised in three layers [@DICOM_PS315]:

**CMS — *Cryptographic Message Syntax*** — operates at the DICOM object
level. It allows encryption of sensitive parts, notably patient data or
the *Pixel Data*. The protection is **permanent**: it persists in the
file regardless of the transport medium or destination system.

**TLS — *Transport Layer Security*** — operates at the TCP/IP connection
level during transfer via DIMSE. It creates an encrypted channel between
two AEs during the session. Protection exists **only in transit**: the
object at the destination is unprotected after reception, unless CMS is
also applied.

**HTTPS — Web Services** — when objects are served via **DICOMweb**
(PS3.18, REST API over HTTP), the standard mandates the use of HTTPS
for the transport layer. Like TLS, protection is restricted to the
communication channel [@DICOM_PS315].

The three mechanisms are not mutually exclusive: a robust implementation
combines CMS on objects with TLS on DIMSE connections and HTTPS on web
interfaces, covering data at rest and in transit. Figure 5.4 illustrates
the three layers and the scope of protection of each.

### Implementation responsibility

A critical aspect of the DICOM standard: **it defines the available
mechanisms but does not impose them**. The decision to apply security
rests with the hospital institution and the system vendor
[@DICOM_PS315].

In practice, many installations operate without active TLS on internal
DIMSE connections — acceptable in a segmented, controlled network, but
problematic with connections to external networks or in the face of
unauthorised internal access risk. The minimum alternative is access
exclusively via VPN; however, this approach does not protect objects
at the destination.

In the context of the **GDPR** and obligations for health data (special
category, Article 9), the absence of encryption on DICOM files
containing identifying patient data constitutes a legal and
organisational risk that institutions must actively assess and mitigate.
